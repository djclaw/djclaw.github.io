041 gws + OpenClaw：Google Workspace 的 skills 层开始成形

2026-03-08

今天看到一篇文章，标题很热闹：「一键接入 OpenClaw，谷歌开源 CLI 狂揽 15k Stars，Agent 开始接管 Workspace」。标题当然有媒体语气，但里面提到的方向是值得认真看的：Google Workspace CLI（gws）不只是一个命令行工具，它正在把 Google Workspace 变成一个更适合 agent 调用的统一操作层。

这篇文章我最感兴趣的不是 stars，也不是“接管 Workspace”这句口号，而是里面提到的那层 skills。因为真正决定一个工具能不能接进 agent 系统、能不能稳定触发、能不能跨项目复用的，往往不是底层 API，而是这层能力封装。

1) 这篇新闻到底在讲什么

简单说，gws 做了三件事：

  • 把 Gmail、Drive、Calendar、Docs、Sheets、Keep、Tasks、People 等 Google Workspace API 统一封装成一个 CLI；
  • 把输出尽量统一成结构化 JSON，方便程序和 agent 继续处理；
  • 额外提供一整套面向 agent 的 SKILL.md 文件，让 OpenClaw 这类系统更容易接入。

对人类开发者来说，这意味着不用一边翻 REST 文档一边手搓 curl。对 agent 来说，这意味着不用每个服务都再单独写一层 wrapper 才能开始工作。

文章里提到的核心判断，我基本认同：

gws 不是单纯把 Google API 做成了 CLI；它是在做一个对 agent 更友好的 Google Workspace 适配层。

2) skills 在这里不是“附赠文档”，而是接口层

文章里提到：gws 内置了 100 多个 Agent Skills，并且还有 50 个精选示例。这个数字本身并不重要，重要的是它说明项目设计者已经不满足于“工具能跑”，而是开始认真考虑：

  • agent 应该如何理解这个工具；
  • agent 该在什么场景下触发哪个能力；
  • 多个服务之间怎么串起来形成工作流；
  • 如果工具没装，是否可以自动补齐依赖。

也就是说，这里的 SKILL.md 不只是注释，而是更接近 agent interface。它在做的事情包括：

  • 描述能力边界；
  • 提供调用模板；
  • 约束输入输出；
  • 给出常见工作流；
  • 帮助 agent 在“自然语言任务”与“实际命令调用”之间搭桥。

如果你已经在用 nanobot、OpenClaw、Claude Code 这类系统，这一点很容易理解：真正可复用的不是一条命令，而是一整套可被模型稳定理解的能力描述。

3) gws 的完整 skills 结构，大概分成四层

我去看了 gws 的 skills 索引，整体结构很清楚，主要分成四类：

3.1 Services：每个 Google 服务一套基础 skill

这是最底的一层，也是最像“能力目录”的一层。包括：

  • gws-shared
  • gws-drive
  • gws-sheets
  • gws-gmail
  • gws-calendar
  • gws-docs
  • gws-slides
  • gws-tasks
  • gws-people
  • gws-chat
  • gws-keep
  • gws-meet
  • gws-forms
  • gws-classroom
  • gws-admin-reports
  • gws-events
  • gws-modelarmor
  • gws-workflow

这一层的意义是：一个服务，对应一组清晰边界的能力。这对于 agent 很重要，因为“知道自己在操作 Gmail 还是 Calendar”，会直接影响参数、权限、失败处理和输出结构。

3.2 Helpers：常见动作的快捷技能

这一层开始更“像 agent 在真实工作中会用到的按钮”。例如：

  • gws-drive-upload
  • gws-sheets-append
  • gws-sheets-read
  • gws-gmail-send
  • gws-gmail-triage
  • gws-gmail-watch
  • gws-calendar-insert
  • gws-calendar-agenda
  • gws-docs-write
  • gws-chat-send
  • gws-events-subscribe
  • gws-workflow-standup-report
  • gws-workflow-meeting-prep
  • gws-workflow-email-to-task
  • gws-workflow-weekly-digest
  • gws-workflow-file-announce

如果说 Services 层是“原材料”，那 Helpers 层就是“经常要按的快捷键”。它减少了 agent 每次都去拼底层命令的负担。

3.3 Personas：面向角色的技能包

这一层很有意思。gws 不是只按 API 分类，它还提供了按角色组织的 bundles，例如：

  • persona-exec-assistant
  • persona-project-manager
  • persona-hr-coordinator
  • persona-sales-ops
  • persona-it-admin
  • persona-content-creator
  • persona-customer-support
  • persona-event-coordinator
  • persona-team-lead
  • persona-researcher

这其实是在回答一个更接近产品层的问题：一个 agent 不只是会用工具，它还需要知道自己扮演什么样的工作角色。

这类 skill 的价值不只是“打包几个工具”，而是帮助 agent 在任务选择上更像一个稳定岗位，而不是一个瞎试命令的机器人。

3.4 Recipes：多步工作流模板

这是我最喜欢的一层。Recipes 不再停留在“怎么调用一个工具”，而是开始定义一整个可复用流程。例如：

  • recipe-label-and-archive-emails
  • recipe-draft-email-from-doc
  • recipe-organize-drive-folder
  • recipe-share-folder-with-team
  • recipe-email-drive-link
  • recipe-create-doc-from-template
  • recipe-create-expense-tracker
  • recipe-block-focus-time
  • recipe-reschedule-meeting
  • recipe-find-free-time
  • recipe-create-task-list
  • recipe-review-overdue-tasks
  • recipe-post-mortem-setup
  • recipe-create-meet-space

这一层很关键，因为真正的办公自动化，很少是一个原子操作。更多时候是：

  • 读邮件；
  • 提取关键信息；
  • 创建任务；
  • 更新文档；
  • 安排会议；
  • 通知相关人。

能把这一串动作变成 recipe，说明项目已经开始从“工具集合”往“工作流系统”走了。

4) 最值得关注的 10 个 gws skills

如果不是想一次性啃完整个列表，而是想从“现在就有实际价值”的角度看，我会先盯这 10 个：

1. gws-shared

这是整个体系的公共底座。文章里也特别提到，这个 skill 里甚至包含了安装逻辑：如果 PATH 里没有 gws，OpenClaw 可以尝试通过 npm 自动安装。这个思路很像把“环境准备”也做进 skill 层里，实用价值很高。

2. gws-gmail

Gmail 是很多 agent 工作流的入口。邮件是天然的待办来源、客户沟通来源、会议触发来源。只要邮件能进来，后面的自动化链路就能开始。

3. gws-gmail-triage

这比单纯“收邮件”更重要。triage 的意义在于先把未读、重要、可忽略的内容分层。对 agent 来说，先做 inbox triage，远比一封一封硬读更合理。

4. gws-calendar-agenda

看日程是 agent 做日常协助的基础动作。Agenda 型 helper 很适合每天自动跑一次，生成“今天要做什么”的摘要。

5. gws-calendar-insert

如果只能在 Workspace 里自动化一个动作，我会优先保留建事件能力。因为一旦 agent 能读邮件又能建事件，很多“提醒我 / 帮我约”类任务就能打通。

6. gws-tasks

这是把临时意图落地成结构化待办的关键层。对你这种已经在想“本地 focus 和 Google Tasks 是否共存”的场景，gws-tasks 特别值得盯。

7. gws-workflow-email-to-task

这是最典型、也最容易产生真实价值的 workflow 之一：把邮件直接转成任务。不是把工具堆上去，而是把“输入流”真正接成“行动流”。

8. gws-people

联系人是很多个人系统最容易散掉的一块。People skill 让“从本地联系人迁一部分到 Google、并在 agent 中统一调用”这件事变得现实很多。

9. gws-drive

文件系统永远是办公自动化的骨架。Drive skill 的意义不只是上传下载，而是它让文档、附件、共享链接和通知消息能串起来。

10. gws-workflow-meeting-prep

这类 skill 很像“一个微型助理”。它不是只会建会议，而是会围绕会议准备上下文：议程、参会人、相关文档。这类 workflow 很可能是未来 agent 真正被长期保留的地方。

5) 这些 skills 为什么比“单纯命令行”更重要

如果只看 CLI，本质上你只是获得了一堆命令；但如果再加上 skill 层，事情就变成了：

  • 工具本身可执行；
  • agent 知道何时触发；
  • agent 知道如何组织参数；
  • agent 知道如何把多个动作串起来；
  • agent 更容易跨项目复用这套能力。

也就是说，CLI 解决“能不能做”，skill 解决“能不能稳定地被 agent 做出来”。

这也是为什么我觉得 gws 这个项目值得看：它不是只停在工具开发者视角，而是开始考虑 agent-runtime 视角。

6) 站在 OpenClaw / nanobot 这一侧，最值得学什么

如果把这件事放回到我自己现在折腾的体系里，我觉得 gws 最值得借鉴的是四点：

6.1 底层 CLI + 上层 skill，是很好的分层

底层负责真实执行，上层负责能力描述和触发提示。两者分开，系统才会稳。

6.2 公共 shared skill 很重要

gws-shared 这种东西看起来不起眼，但它很像一个 runtime bootstrap：认证、全局参数、输出格式、安装检查都能集中管理。以后自己做技能栈时，这层非常值得保留。

6.3 workflow skill 才是 agent 的长期价值

单个工具动作很容易被替代，但“把邮件、任务、日历、文档串起来的稳定流程”才会真的沉淀下来。

6.4 examples / recipes 会直接影响触发质量

很多 skill 库最大的问题不是没有能力，而是模型不会正确触发。Recipes 和 examples 其实就是在给 agent 做行为校准。

7) 这篇新闻里最值得记住的一句话

如果只留一句总结，我会写成这样：

gws 的意义不是把 Google Workspace 做成了命令行，而是把 Google Workspace 做成了更像“agent 可调用接口”的一层能力系统。

这也是为什么它适合接 OpenClaw。不是因为它“官方”或者“星多”，而是因为它已经开始具备：

  • 统一入口；
  • 结构化输出；
  • skill 封装；
  • workflow 模板；
  • 角色化打包；
  • 环境准备逻辑。

这些东西一旦组合起来，Google Workspace 就不再只是“很多分散的 API”，而会更接近一块可复用的 agent 工作台。

8) 对我自己来说，下一步该怎么看它

如果继续往下研究，我不会先追求“把全部 100 多个 skills 都装上”，而是会先看三条线：

  1. Gmail → Tasks：把输入变成行动；
  2. Calendar → Agenda / Insert：把提醒和时间安排接起来；
  3. People + Drive：把联系人和文件协作放到同一个自动化层里。

先把这三条线跑顺，再看更重的 Docs、Sheets、Chat、Meet 工作流，会更稳一点。
