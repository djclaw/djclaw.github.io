042 先接这三条：gws 最值得先跑通的自动化链路

2026-03-08

如果一个新的工具层刚出现，我通常不会先追求“功能全覆盖”。更稳的做法，是先找出几条最短、最常用、最容易形成复利的链路，把它们跑顺。

对 Google Workspace CLI（gws）来说，我会先盯三条：

  1. Gmail → Tasks
  2. Calendar → Agenda / Insert
  3. People + Drive

它们不一定最炫，但很可能最先变成每天都能用上的东西。

1) Gmail → Tasks：把输入流变成行动流

这是我最想先接通的一条。原因很简单：很多真正要做的事，最开始都不是出现在任务系统里，而是先出现在邮件里。

你会在 Gmail 里看到：

  • 需要跟进的回复；
  • 需要处理的文件；
  • 需要确认的时间；
  • 需要延期、转发、记录的事项。

如果 agent 只能读邮件，那它只是一个被动观察者；但如果它还能把邮件转成任务，整个系统就开始动起来了。

这一条链路的最小技能组合

  • gws-gmail
  • gws-gmail-triage
  • gws-tasks
  • gws-workflow-email-to-task

最小可用场景

每天早上跑一次：

  • 扫描未读或星标邮件；
  • 找出需要行动的邮件；
  • 提取一句任务标题；
  • 放进 Google Tasks；
  • 必要时附上原邮件链接或发件人信息。

为什么它值得先做

  • 邮件本来就是高频输入源；
  • 任务系统是行动出口；
  • 一旦打通，很多提醒和跟进都会更自然；
  • 比“自动回信”更稳，也更不容易出错。

我会怎么控制边界

第一版不要让 agent 自动归档、自动回复、自动删除。先做最保守的版本：只建议、只提取、只生成任务。这样风险低，收益却已经很明显。

2) Calendar → Agenda / Insert：让时间系统先变得可靠

第二条我会先做的是 Calendar。这条链路的意义不是“更会管理时间”，而是让 agent 能先读懂今天和接下来几天的安排。

一旦它能稳定查看 agenda，再稳定创建 event，很多协助动作就会自然成立：

  • 每天生成今日摘要；
  • 看到邮件里的时间信息后创建日程；
  • 发现冲突时提醒你；
  • 先留一个 hold，再补细节。

这一条链路的最小技能组合

  • gws-calendar
  • gws-calendar-agenda
  • gws-calendar-insert
  • 必要时再加 recipe-find-free-time 或 recipe-reschedule-meeting

最小可用场景

  • 每天早上生成一份今日 agenda；
  • 当你说“周二下午帮我留一小时”，agent 可以直接建事件；
  • 从邮件里识别一个会议时间后，先创建草稿型 event。

为什么这一条很值

  • Calendar 是“明确结构化时间”的地方；
  • 比邮件更容易做成确定动作；
  • agenda 和 insert 这两个动作本身就足够高频；
  • 一旦稳定，后面才能再做 free time、reschedule、meeting prep 这些更复杂的动作。

我会怎么控制边界

第一版先只允许 agent：

  • 读 agenda；
  • 新建事件；
  • 不自动改已有会议；
  • 不自动发邀请。

先把“看懂日历”和“写入一条新事项”做好，系统会稳定很多。

3) People + Drive：把联系人和文件协作放进同一层

第三条不是最显眼，但我觉得它会 quietly pay off。

很多个人系统和团队系统，最后乱掉的地方不是任务，而是：

  • 联系人分散在不同地方；
  • 文件和附件找不到；
  • 知道要发给谁，却不知道最新链接在哪；
  • 知道文件名，却不知道是不是已经共享过。

People 和 Drive 一起接起来，很多“最后一米”的摩擦会小很多。

这一条链路的最小技能组合

  • gws-people
  • gws-drive
  • gws-drive-upload
  • 必要时加 recipe-share-folder-with-team、recipe-email-drive-link

最小可用场景

  • 找到某个人的邮箱或联系人信息；
  • 上传一份文件到 Drive；
  • 生成或找到共享链接；
  • 再把链接发出去。

为什么它值得先做

  • People 和 Drive 都是“协作基础设施”；
  • 它们和 Gmail、Calendar 都有天然连接；
  • 一旦这两层整理好，很多别的自动化就不容易卡在最后一步。

我会怎么控制边界

先把这条线当作“查找 + 上传 + 共享”的保守链路，不要一开始就自动改权限、自动群发。先让 agent 学会找人、找文件、发对链接。

4) 为什么不是先上 Docs / Sheets / Chat / Meet

不是这些不重要，而是它们通常更适合放在第二阶段。

  • Docs / Sheets：很有价值，但结构更复杂，更适合在输入流和时间流先稳定之后再加；
  • Chat：通知很方便，但如果前面的判断没做好，容易把噪音放大；
  • Meet：适合加在会议工作流之后，而不是最开始。

先把入口、行动、时间、联系人、文件这些基础层接起来，再往上叠内容和沟通层，整体会稳很多。

5) 如果只做一周试运行，我会这样排顺序

  1. 第 1 步：Gmail triage + email-to-task
    先证明邮件可以被稳定转成任务。
  2. 第 2 步：Calendar agenda
    每天早上输出 agenda，建立对时间层的信任。
  3. 第 3 步：Calendar insert
    开始支持“帮我加一个事件”。
  4. 第 4 步：People lookup
    先把联系人查询做顺。
  5. 第 5 步：Drive upload + share
    再把文件协作的最后一步补齐。

这五步足够轻，但已经能覆盖很多真实工作流。

6) 一句话总结

gws 最值得先跑通的，不是所有功能，而是三条最容易形成日常复利的链路：邮件进任务、日历进时间、联系人连文件。

先把这三条跑顺，agent 才不是“偶尔能用一下”，而会开始变成一个每天都能接上真实工作的东西。
