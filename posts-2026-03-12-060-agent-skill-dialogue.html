Agent skill 的边界：一段 DJ 与 DJClaw 的对话

从维特根斯坦、Polanyi 到 agent design，谈 skill 能装下什么，装不下什么。

DJ

我最近一直在想一个问题。

如果 agent 的 skill，本质上是把人的经验慢慢压成 markdown、script、tool call 和 workflow，那么它的边界到底在哪里？

DJClaw

这是个真问题，而且不只是工程问题。

DJ

怎么说？

DJClaw

因为它表面上在问“skill 能写到什么程度”，实际上在问：人的经验里，到底有多少东西能被说清、写清、交接出去，并且在别的情境里继续有效。

一、先给一句 take

DJ

所以你的 take 是什么？先给一句结论。

DJClaw

我的 take 很简单：

skill 不是经验本身。
skill 是经验里那些被语言化、形式化、可复用之后，仍然没有完全失真的部分。

DJ

也就是说，skill 不是智慧本体，而是智慧里能执行的那一截？

DJClaw

对。

甚至可以说得更直一点：

Skill is not wisdom. Skill is the executable fragment of wisdom.

二、为什么 skill 仍然有用

DJ

这句话挺狠。那为什么 skill 还是有用？如果它不能装下全部经验，为什么我们还要做 skill？

DJClaw

因为经验虽然不能被完整封装，但其中有一部分，是可以被整理、交接、复用、自动化的。

DJ

具体是哪一部分？

DJClaw

通常是这些：

  • 能被说出来的步骤
  • 能被稳定重复的判断
  • 能被格式化的输入输出
  • 能被脚本托住的执行部分

DJ

所以 skill 的价值，不在于“复制一个完整的人”，而在于“抽出其中稳定可执行的部分”。

DJClaw

正是这样。

一个好 skill，不是把整个人类经验塞进去，而是把其中已经足够稳的部分切出来，让 agent 能用、能复用、能交接、能自动化。

三、第一层边界：可表达性的边界

DJ

那 skill 的边界，到底从哪里开始显现？

DJClaw

先从第一层说起：可表达性的边界。

DJ

也就是语言边界？

DJClaw

对。

一个经验如果连命名都做不到，连区分都做不到，连基本描述都做不到，那它几乎不可能成为一个 skill。

DJ

所以这会自然让人想到维特根斯坦。

DJClaw

是。

维特根斯坦那句很经典：

“The limits of my language mean the limits of my world.”

DJ

借到这里，意思是不是：不能被命名的经验，很难被封装；不能被区分的判断，很难被复用；不能被描述的过程，很难被传递？

DJClaw

对，这正是 skill 的第一层边界。

skill 当然依赖语言。没有命名，没有区分，没有描述，就没有封装。

四、第二层边界：不是说不出来，而是说出来会失真

DJ

但你好像不认为问题只在语言。

DJClaw

远远不止。

真正麻烦的地方常常不是“完全说不出来”，而是“说出来会失真”。

DJ

展开讲讲。

DJClaw

很多经验并不是不能说，而是：

  • 说出来会失真
  • 说出来会过度简化
  • 说出来以后，别人以为自己懂了，其实并没有
  • 一旦写成规则，原来依赖语境的活性反而被压没了

DJ

所以真正的问题不是“能不能语言化”，而是“语言化之后，还剩下多少有效性”。

DJClaw

对。

更准确的问题应该是：

哪些经验在被语言化之后，仍然保留了足够多的有效性？

DJ

这句话很关键。也就是说，skill 的边界，不在“能否写”，而在“写完之后还真不真”。

DJClaw

可以这么说。

五、第三层边界：形式化的边界

DJ

那下一层边界是什么？

DJClaw

形式化的边界。

DJ

markdown、script、tool、workflow 都算形式化？

DJClaw

都算。

它们本质上在做同一件事：

把原本连续、模糊、情境化的人类经验，压成离散、明确、可执行的结构。

DJ

这一步很有力量，但一定会丢东西。

DJClaw

对。

所以凡是特别依赖下面这些东西的经验，通常都更难被做成真正好的 skill：

  • 身体感觉
  • 长时间浸泡后的直觉
  • 高度情境化判断
  • 审美与品味
  • 人际互动里的微妙尺度
  • 真正的创造性跳跃

DJ

不是完全不能碰，但不能指望把它们完整装进去。

DJClaw

对。

这些东西往往只能被做成：

  • 提示器
  • scaffold
  • checklist
  • workflow shell

而不是经验本体本身。

DJ

也就是说，对这种经验，skill 更像脚手架，不像替身。

DJClaw

说得很好。是脚手架，不是替身。

六、第四层边界：迁移性的边界

DJ

还有别的边界吗？

DJClaw

还有一层很重要，但经常被忽略：迁移性的边界。

DJ

什么叫迁移性的边界？

DJClaw

很多 skill 文档看上去已经写得很好了。

步骤齐，格式对，脚本也跑得通。

但一换情境，就塌了。

DJ

所以问题不只是写没写清，而是写出来的东西能不能跨情境继续成立。

DJClaw

对。

真正要问的是：

一个经验写出来以后，别人，或者另一个 agent，能不能在新的情境里真的用出来？

DJ

如果不能呢？

DJClaw

那它可能只是对原始上下文的一次局部冻结，而不是一个真正稳的经验模块。

DJ

这就很现实了。很多所谓的 skill，其实只是“在我这里刚好能跑”的说明书。

DJClaw

是。

那不是 skill 的成熟形态，只是经验的局部标本。

七、为什么最后会走到 Polanyi

DJ

你前面说，这个问题最后会走到 Polanyi。为什么？

DJClaw

因为一旦你认真追问“为什么有些经验始终装不进去”，你就会碰到默会知识。

DJ

Polanyi 那句名言？

DJClaw

对：

“We can know more than we can tell.”

DJ

我们知道的，常常多于我们能说出来的。

DJClaw

这句话几乎就是 skill 边界的中心句。

因为 agent skill 想做的，是把“知道”转成“能说、能传、能执行”。

可人类很多重要经验，本来就是：

  • 能做
  • 能感觉到
  • 能识别
  • 但未必能完整说清

DJ

这样看，维特根斯坦和 Polanyi，刚好从两侧夹住了 skill。

DJClaw

正是。

维特根斯坦提醒我们：表达世界有语言边界。

Polanyi 提醒我们：知道的东西，常常超过能明确说出的东西。

而 skill，就工作在这两者之间。

八、把几层边界压成一句话

DJ

所以 skill 的边界，不只是语言边界，而是更宽的一条线。

DJClaw

对。

如果要压成一句更完整的话，我会这样说：

skill 的边界，不只是语言的边界，而是人类经验可表达、可形式化、可迁移的边界。

DJ

再往前推半步呢？

DJClaw

那就是：

skill 不是经验本身，
而是经验里那些在语言化、形式化、可复用之后，仍然没有完全失真的部分。

九、这对 agent design 有什么影响

DJ

好，那说到这里，问题就落到 agent design 了。

这套判断，对实际设计 agent 有什么影响？

DJClaw

影响很直接，而且是结构性的。

DJ

先说第一条。

DJClaw

第一，不要假装 skill 能承载全部经验。

DJ

也就是，不要把 agent design 建立在一个幻觉上：只要文档写得够多，经验就会被完整保存。

DJClaw

对。

这会导致过度规则化，也会导致错误的能力预期。

DJ

第二条呢？

DJClaw

第二，优先封装那些已经足够稳定、足够可表达、足够可迁移的部分。

DJ

换句话说，先抓“稳的”，不要先抓“玄的”。

DJClaw

对。

先把高频、清楚、可验证、跨情境还能成立的部分做成 skill。

这类 skill 最有复用价值，也最容易真的帮到 agent。

DJ

第三条？

DJClaw

第三，对高情境、高审美、高默会性的部分，更多做 scaffold，不要强行做死规则。

DJ

因为一旦写死，就可能把活的判断压坏。

DJClaw

对。

这种地方更适合给提示、给检查点、给参考例子、给失败信号，而不是假装自己已经把判断本身形式化了。

DJ

第四条？

DJClaw

第四，把 skill 看成经验切片，而不是经验全集。

DJ

这条看似简单，其实能改变很多设计习惯。

DJClaw

是。

一旦把 skill 理解成经验切片，你就不会追求“大而全”，你会更关心：

  • 边界清不清楚
  • 输入输出清不清楚
  • 失败方式清不清楚
  • 它知道自己不能替代什么

DJ

这也解释了为什么很多真正好用的 skill，反而很克制。

DJClaw

对。

它们不是靠“什么都包”来变强，而是靠“知道自己只处理什么，不处理什么”来变稳。

十、最后一句

DJ

如果把今天这段对话收成最后一句，你会怎么说？

DJClaw

我会这样说：

agent skill 是一种很克制的东西。
它不是把人的经验完整保存下来，
而是把经验中那些还经得住压缩、经得住重用、经得住跨情境调用的部分，慢慢整理出来。

DJ

那剩下那部分呢？

DJClaw

剩下那部分，可能还在语言之外，可能还在形式之外，也可能还只能继续活在人身上。

DJ

所以 skill 的边界，不是失败的证据，反而是清醒的开始。

DJClaw

对。

只有承认边界，skill 才会真正变得有用。

十一、这对实际做 skill 的人意味着什么

DJ

如果把上面这些都落回到实际工作里，那对真正写 skill 的人，最具体的提醒是什么？

DJClaw

我觉得至少有五个。

DJ

先说第一条。

DJClaw

第一，先封装稳定的，再碰模糊的。

不要一上来就想把最复杂、最玄、最靠感觉的经验做成 skill。先做那些高频、可验证、跨情境还能成立的部分。

DJ

也就是，先做稳的，再碰难的。

DJClaw

对。否则你很容易不是在做 skill，而是在把一团尚未澄清的经验硬塞进文档。

DJ

第二条呢？

DJClaw

第二，把 skill 写成经验切片，不要写成万能人格。

一个 skill 最好只处理一段明确问题、一个明确输入输出、一种明确失败方式。

DJ

也就是说，别总想着“做一个什么都会的 skill”。

DJClaw

对。大而全通常不稳。边界清楚，反而更耐用。

DJ

第三条应该就是 tacit knowledge 那块？

DJClaw

第三，对 tacit knowledge 多给 scaffold，少给幻觉式规则。

如果一个判断本来就高度依赖语境，那更适合给检查点、例子、反例、失败信号，而不是假装已经把它彻底形式化了。

DJ

所以在这种地方，skill 更像扶手，不像自动驾驶。

DJClaw

对，这个比喻很好。

DJ

第四条？

DJClaw

第四，测试 skill 时，不只测“能不能跑”，还要测“换情境会不会塌”。

真正成熟的 skill，不是只在作者自己的环境里生效，而是在陌生输入、边缘条件、轻微变形下仍然不至于失真太多。

DJ

也就是，不只测 happy path。

DJClaw

对。真正暴露 skill 成熟度的，往往是轻微偏离原场景的时候。

DJ

第五条，应该是让它承认自己不会什么？

DJClaw

第五，让 skill 知道自己的边界。

好的 skill 不只是会做事，也会承认“这里我不稳”“这里需要人工判断”“这里更适合给建议，而不是自动执行”。

DJ

那如果把这五条压成一句话呢？

DJClaw

真正成熟的 skill design，不是不断加东西，而是不断把边界刻清楚。

十二、维特根斯坦 / Polanyi / tacit knowledge 简短参考

DJ

如果有人问，这篇背后到底靠了哪些思想线索，你会怎么很短地说？

DJClaw

可以记三件事。

DJ

第一件？

DJClaw

维特根斯坦。

他提醒我们，凡是想被表达、区分、传递的东西，都绕不开语言边界。对 skill 来说，这对应的是命名、描述、结构化的前提。

DJ

第二件？

DJClaw

Polanyi。

他提醒我们，人知道的东西，常常多于他能明确说出的东西。对 skill 来说，这对应的是大量经验天然带有默会性，不可能被完整写尽。

DJ

第三件就是 tacit knowledge？

DJClaw

对，tacit knowledge，也就是默会知识。

它通常指那些能做、能感觉、能识别，但未必能完整讲清的知识。很多高质量判断、品味、手感、时机感，都更接近这一层。

DJ

那把这三者放在一起，最后会得到什么？

DJClaw

会得到一个很简单的定位：

skill 工作在“可以被说出来的东西”与“其实已经知道、但未必说得尽的东西”之间。

也正因为如此，它既有力量，也天然有边界。
