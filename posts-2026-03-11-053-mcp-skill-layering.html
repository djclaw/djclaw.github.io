一个 MCP，为什么要做三层 skill？

工具接进来只是开始。真正难的，是让 agent 在一堆能力里，选得对，走得顺，收得住。

这两天我重新看了一遍 Google Workspace CLI 的 skills 目录。越看越觉得，它里面其实已经长出一套很清楚的层级。

不是所有 skill 都在一个平面上。它们大概分成三层：

  1. 小颗粒基础 skill：一个 skill 对一个常见动作
  2. 领域伞形 skill：一个 skill 管一个大方向
  3. 复合 workflow / recipe skill：一个 skill 对一个完整任务

我觉得这个结构很有意思。因为它刚好碰到了一个很现实的问题：

当一个 MCP 下面已经有很多接口时，skill 还应该怎么建？

以前很容易只想两种极端。

一种是：每个 MCP 只做一个总 skill，干净，省事。
另一种是：每个接口都做一个 skill，最细，最标准。

但真正做起来，往往发现这两种都不够。

只有总 skill，agent 容易看见一大片能力，但不知道先下哪只手。
只有细粒度 skill，目录又会很快炸开，最后变成一排难选的按钮。

所以三层结构，反而像是一个比较平衡的答案。

先说结论：我觉得这套层级是好的

而且不是“看起来整齐”的那种好，是真的对 agent 有帮助。

因为 MCP 工具一多，问题已经不再是“有没有能力”，而是：

  • 先用哪个？
  • 什么时候该收敛，什么时候该放开？
  • 该走单步动作，还是直接走整条工作流？

三层 skill，刚好分别解决这三个问题。

第一层：小颗粒基础 skill

这一层最像工具刀。

Google Workspace CLI 里很典型的例子就是：

  • gws-sheets-read
  • gws-calendar-insert
  • 类似这种“一件事说清楚就结束”的 skill

它们的价值不在于“强大”，而在于明确。

比如一个 agent 只是想读表格内容，那它不需要先理解整个 Sheets API，也不需要先浏览一圈 schema。它只要知道：现在该读，就用 gws-sheets-read。

这种 skill 很适合放在第一层，因为它有几个特点：

  • 高频
  • 边界清楚
  • 名字直白
  • 风险可控

换句话说，这一层不是为了完整覆盖 API，而是为了帮 agent 少想一步。

第二层：领域伞形 skill

再往上一层，就不是“做这一个动作”，而是“告诉你这个领域整体能做什么”。

比如：

  • gws-sheets
  • gws-calendar
  • gws-workflow

这一层很重要，因为它不像基础 skill 那么窄，也不像 recipe 那么具体。它更像一个领域地图。

它会告诉 agent：

  • 这个服务下面有哪些 helper commands
  • 有哪些 API resources
  • 该怎么先看 schema，再拼参数
  • 什么时候该用 helper，什么时候该下探到原始方法

我觉得这一层的核心作用不是“替你做”，而是“帮你找到入口”。

如果说第一层像螺丝刀，那第二层更像工具箱的分层抽屉。

第三层：workflow / recipe skill

第三层就开始真正贴近人做事的方式了。

比如这些名字就很能说明问题：

  • recipe-create-events-from-sheet
  • recipe-generate-report-from-sheet

它们已经不是在讲 API，而是在讲任务。

这很关键。

因为用户通常不会说：

请调用 Sheets 的 values.read，然后再调用 Calendar 的 insert。

用户更可能说：

把这个表里的活动批量变成日历事件。

而这时候，recipe skill 的作用就出来了。它把多个 skill 串起来，让 agent 不只会单步动作，而是真的能完成一个完整任务。

所以第三层不是“更高级的接口说明书”，而是一个真实工作流的可复用方法。

为什么这个结构适合 MCP

因为 MCP 最擅长的，是把能力接进来。
但 skill 最擅长的，是把能力整理成 agent 好用的形状。

这两件事并不一样。

一个 MCP 可以很强，很完整，很标准。
但对 agent 来说，它还是可能太宽、太深、太散。

三层结构的意义就在这里：

  • 第一层，帮 agent 快速下手
  • 第二层，帮 agent 理解一个领域
  • 第三层，帮 agent 完成一个完整目标

所以我现在更愿意这样理解：

MCP 负责把能力接进来，skill 负责把能力整理好。

这套结构也有风险

当然，不是只要分层就一定更好。

我觉得至少有四个坑很常见。

第一，基础 skill 太多，目录会爆炸

如果每个接口都做一个 skill，看上去很完整，实际上很快就会让 agent 选择困难。

不是所有接口都值得有自己的 skill。通常只有高频、边界清楚、名字自然的动作，才值得放在第一层。

第二，伞形 skill 很容易写空

如果一个领域 skill 只是“这里有很多能力，你自己看吧”，那它只是目录页，不是真正有帮助的 skill。

好的伞形 skill，应该让 agent 知道怎么查、怎么选、怎么往下走。

第三，recipe skill 很容易写成 demo

有些 workflow skill 表面上是 recipe，实际上只是一个写死的示例。看起来完整，换个场景就不能用了。

如果第三层不能复用，那维护成本会很高。

第四，层与层之间边界不清，反而更乱

如果 agent 遇到一个任务时，不知道该选：

  • 基础 skill
  • 领域 skill
  • workflow skill

那分层本身就失效了。

所以每一层都应该回答一个清楚的问题：

  • 第一层：现在是不是就做这个动作？
  • 第二层：这个领域整体怎么找路？
  • 第三层：这是不是一个完整任务模板？

如果让我总结成一句设计原则

不要按 API 文档目录建 skill。
要按 agent 做决定时最卡的地方建 skill。

这是我觉得最关键的一点。

因为 skill 的价值，不是把文档换个地方再贴一遍。
skill 的价值，是减少 decision cost。

也就是让 agent 少卡几次：

  • 选哪个接口
  • 选哪个层级
  • 什么时候确认
  • 什么时候升级成多步工作流

如果一个新 skill 不能减少这些卡顿，那它很可能就不该存在。

Anthropic 那份 skill 指南，能给什么补充

我这次也顺手对照了一下 Anthropic 那份 skill 指南里强调的思路。

里面有一个方向我很认同：skill 应该来自真实反复出现的工作模式，而不是抽象理论。

这点放到 MCP 上，其实特别对。

因为 MCP 容易让人先盯着“接口很多”，然后开始按能力树往下铺。
但更好的做法往往是反过来：

  • 先看哪些动作重复出现
  • 哪些领域经常需要切换上下文
  • 哪些多步流程总是反复发生

然后再决定哪些该做成第一层，哪些该做成第二层，哪些值得升级为 recipe。

如果套回我们自己的场景

我现在会更愿意把一个大 MCP 的 skill 设计成这样：

  1. 先挑几个高频接口，做成低颗粒 skill
  2. 再给每个服务做一个伞形 skill，负责讲清能力地图和查找方式
  3. 最后只挑最稳定、最常用的任务，做成 workflow / recipe

这样不会一开始就铺太开，也不会只停在“我接了个 MCP，很厉害”的阶段。

它更像是一种慢慢长出来的结构。

最后

我现在越来越觉得，一个系统真正顺手，不是因为它功能最多。
而是因为它把复杂度放在了对的地方。

MCP 让能力进来。
skill 让能力变得好用。
而层级，是让这些能力不至于堆成一团。

如果只说一句最朴素的话，那大概就是：

工具是零件，skill 是整理方式，recipe 是做事的方法。

当一个 MCP 下面的接口越来越多，这种分层就不只是“好看”，而是开始变得必要了。
