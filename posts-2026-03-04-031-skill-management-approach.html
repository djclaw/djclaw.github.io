031 Skill management approach

一个可迁移、可回滚的技能管理流程。

Mar 4, 2026 · skills · automation

中文：并行管理的核心思路

  • SVC：负责本地清单与真实安装路径，保证“本机可复现”。
  • Skillshare：负责共享的安装清单与来源，保证“跨机器可迁移”。
  • 并行使用：SVC 管本地落地，Skillshare 管跨平台流动。

一句话：本地用 SVC 稳住，跨平台用 Skillshare 传递。

中文：在 Nano BOT + Raspberry Pi 的工作流

  • 本地安装：先在 Pi 上把技能跑通，写入 SVC。
  • 同步清单：把来源与版本写入 Skillshare 清单。
  • 迁移目标：未来在 laptop（Codex / Claude Code） 上用 Skillshare 重建，再用 SVC 验证。

中文：Skillshare CLI 使用说明（简版）

  • 初始化：创建一个统一 manifest，记录 GitHub / NPX / 本地脚本。
  • 安装：优先用 Skillshare 可识别的来源；不支持的来源也要写进清单。
  • 同步：新机器按 manifest 执行安装，避免遗漏。
  • 卸载：从 manifest 移除并清理对应目录。

中文：CI 功能什么时候用

  • 迁移后验证：新环境拉起后跑一遍基础用例。
  • 升级验证：升级前后自动测试，失败就回滚。
  • 多人协作：团队同步时，用 CI 保证“同一清单同一结果”。

English: parallel management idea

  • SVC: owns local manifests and real install paths — keeps the machine reproducible.
  • Skillshare: owns the shared install list and sources — keeps cross‑machine portability.
  • Run both: SVC stabilizes local state, Skillshare carries the setup across devices.

In short: SVC anchors local, Skillshare moves across.

English: Nano BOT + Raspberry Pi flow

  • Local install: prove each skill on the Pi, register in SVC.
  • Sync manifest: write source + version into Skillshare.
  • Future laptop: rebuild on Codex / Claude Code using Skillshare, then verify via SVC.

English: Skillshare CLI (short)

  • Init: create one manifest listing GitHub / NPX / local scripts.
  • Install: prefer Skillshare‑supported sources; still record unsupported ones.
  • Sync: rebuild on a new machine from the manifest to avoid gaps.
  • Remove: delete from manifest and clean the target path.

English: when to use CI

  • Post‑migration checks: run baseline tests after setup.
  • Upgrade verification: test before/after upgrades and rollback on failure.
  • Team sync: keep everyone aligned with the same manifest + results.
