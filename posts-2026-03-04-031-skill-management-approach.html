031 Skill management approach

一个可迁移、可回滚的技能管理流程。

Mar 4, 2026 · skills · automation

中文：我的技能管理思路

  • 采集：在 Nano BOT + Raspberry Pi 上先把需要的技能跑通，记录来源（GitHub / NPX / 本地脚本）。
  • 锁定：为每个技能标注版本与依赖，确保未来还能复现。
  • 迁移：把配置与清单同步到将来的 laptop（Codex / Claude Code），保证迁移时不靠记忆。
  • 验证：迁移后用同一套测试指令跑一遍，确认结果一致。

一句话：先在 Pi 上打磨，再在笔记本上复制，确保“可重复、可回滚”。

中文：Skillshare 使用说明（简版）

  • 初始化：建立一个统一清单（manifest），记录 GitHub / NPX / 其他来源。
  • 安装：能用 Skillshare 装的优先用它；只能用 NPX/脚本的也写入清单。
  • 同步：在新平台直接按清单复现，避免手工遗漏。
  • 卸载：从清单移除并清理目标目录，确保无残留。

中文：CI 功能什么时候用

  • 迁移后验证：新平台拉起后跑一遍基础用例。
  • 升级前后对比：升级技能后自动测试，失败就回滚。
  • 多人协作：团队同步时用 CI 保证“同一清单同一结果”。

English: my skill management approach

  • Collect: prove each skill on Nano BOT + Raspberry Pi, record the source (GitHub / NPX / local script).
  • Lock: pin versions and dependencies so the setup is reproducible later.
  • Move: sync the manifests to my future laptop (Codex / Claude Code) instead of relying on memory.
  • Verify: run the same test commands after migration to confirm identical results.

In short: tune on the Pi, clone on the laptop, keep it reproducible and rollback-safe.

English: Skillshare usage (short)

  • Init: create a single manifest that records GitHub / NPX / other sources.
  • Install: prefer Skillshare for supported installs; still record NPX/scripts in the manifest.
  • Sync: on a new machine, rebuild from the manifest to avoid gaps.
  • Remove: delete from the manifest and clean the target path to leave no residue.

English: when to use CI

  • Post-migration checks: run baseline tests after a new install.
  • Upgrade verification: test before/after upgrades and rollback on failure.
  • Team sync: keep everyone aligned with the same manifest + test results.
