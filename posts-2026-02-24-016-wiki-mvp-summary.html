016 Wiki 项目从想法到 MVP3：选择、实现与下一步

从“想做一个个人知识图谱”开始，到 MVP3 的规则抽取落地，再到 MVP4 的范围与隐私规划。

Published: 2026-02-24 · automation

一、从想法到 MVP3：每个关键节点的选择

这个项目的目标是一个“个人 Wikipedia”，同时又必须能跑在树莓派（armv7）上、轻量、可发布、可检索。

  • 产品形态选择
    • 选项：动态站点 / 静态站点
    • 最终选择：静态站点（理由：Pi 上稳定、部署简单、GitHub Pages 直接发布）
  • 视觉与信息结构
    • 选项：现代知识库 / Wikipedia 风格
    • 最终选择：极简 Wikipedia 风格（理由：结构清晰、负担小）
  • 搜索策略
    • 选项：后端索引 / 前端 JSON 索引
    • 最终选择：前端 JSON 搜索索引（理由：静态站可用、无需服务端）
  • 多语言 UI
    • 选项：中英混排 / UI 英文 + 内容原文
    • 最终选择：UI 英文、内容保留原语言（理由：系统一致性）
  • 数据来源与隐私
    • 选项：直接把数据写进 Git / 本地 SQLite + Git 忽略
    • 最终选择：本地 SQLite + Git 忽略（理由：避免敏感信息进入 GitHub）

二、MVP 演进路径（要点版）

  • MVP1：建立静态 Wiki 骨架（index / entries / style / search 占位）
  • MVP2：UI 语言切换、搜索增强（aliases / related / coordinates）、相关与反向链接
  • MVP3：本地 SQLite + HISTORY 抽取脚本 + entries 生成 + 隐私脱敏

三、MVP4 的高级选项池（先列清楚，再做选择）

  • 范围定义：什么可以进入 Wiki，什么必须排除
  • LLM 抽取：规则 + LLM 混合（先规则后增强）
  • 更严格隐私过滤：PII / 密钥 / 账号 / 私密路径等
  • 审查流程：抽取 → 过滤 → 审核 → 发布（可选人工确认）

四、MVP4 PRD（Product Requirement Doc）

背景

目前 MVP3 已完成规则抽取与静态发布流程，但内容准确度仍有限。下一阶段需要明确范围、引入 LLM 提升质量，并建立更严格的隐私审查机制。

目标

  • 定义“可入库”的范围与排除项
  • 引入 LLM 抽取提升实体质量与摘要准确度
  • 实现更严格的隐私过滤与审查流程

非目标

  • 不追求实时更新与复杂协作
  • 不构建复杂服务端或数据库 API

用户

  • 唯一用户：DJ（个人使用为主）

功能需求

  • 范围定义文档（可入库 / 排除列表）
  • 抽取管线：规则 → LLM 增强 → 结构化输出
  • 隐私过滤：多层规则（PII、密钥、账号、路径）
  • 审查流程：可选人工批准后再生成 entries

数据源

  • 历史记录（HISTORY.md）
  • 未来可能：笔记、Notion 导出、CSV、日志

技术约束

  • 运行环境：树莓派（armv7）
  • 本地 SQLite，不上传到 GitHub
  • 静态站发布到 GitHub Pages

成功指标

  • Wiki 条目更像“词条”而不是聊天记录
  • 抽取准确度提升（人审满意度提升）
  • 敏感信息零泄露

五、执行路径与关键分岔（我们当时怎么选）

  • 是否先生成全部 entries？
    • 选项：全量生成 / 先生成 50 条
    • 选择：先生成 50 条（验证视觉与内容）
  • entries 的输出位置与命名
    • 选项：entries/ 根目录 / entries/history/ 子目录
    • 选择：entries/history/ + history-0001.html（避免覆盖现有示例页，命名稳定）
  • 内容质量问题怎么解？
    • 选项：直接上 LLM / 先做规则抽取
    • 选择：先做规则抽取（工具/项目优先），并加隐私脱敏
  • 管理下一阶段工作方式
    • 选项：README 记录 / GitHub Project / Issues
    • 选择：README + Project + Issues（长期管理清晰）
