Quiet notes on harness engineering

Less typing. More structure. More judgment about what the agent can actually see.

OpenAI’s piece on harness engineering stayed with me for a simple reason: it does not describe agents as magic. It describes them as systems that need good ground under their feet.

The striking part is not just the scale. A small team used Codex to build a large internal product, with humans steering and agents writing the code. The deeper shift is in what the engineers were actually doing. They were not spending most of their time typing application code by hand. They were building the environment that made reliable work possible.

That means clearer repository structure. Better instructions. Better docs. Better observability. Better feedback loops. Better boundaries. In other words, less heroics inside a prompt, and more care in the system around the prompt.

One idea felt especially important: if the agent cannot see it, it effectively does not exist. Notes in someone’s head, scattered chat messages, hidden assumptions, stale docs — none of that helps much. The repository has to become the place where knowledge lives in a form the agent can navigate.

Another useful point: the goal is not to make everything verbose. It is to make things legible. A short map is often better than one giant manual. A clean architecture is better than a clever one. A strict boundary is often more helpful than an open field.

The article also reframes engineering taste. With agents, taste is not only about elegance in code. It becomes architecture, invariants, checks, naming, docs, and feedback loops. You do not need to micromanage every line if the system makes good lines easier to produce.

I also liked the quiet honesty in the tradeoff: when agent throughput becomes high, some old habits change. Long blocking loops become expensive. Fast correction matters more. Clean merge philosophy matters differently. The bottleneck moves from code-writing to human attention.

So the main takeaway for me is simple.

In an agent-first setup, the work moves upward.

You spend less time writing code directly.

You spend more time designing the terrain.

You spend more time deciding what the agent should see, what it should ignore, and how it should know whether it did a good job.

That feels like a real shift. Not smaller engineering. Just a different layer of it.
