A Logseq-Inspired Kernel for My Wiki

The wiki does not need to become a full Logseq clone. But it can borrow the right inner model: pages, blocks, links, backlinks, timeline, and a small extraction pipeline that turns history into a quiet public knowledge surface.

I have been thinking about whether Logseq should sit under the wiki project as the real engine, or whether it should stay more like a reference point.

The answer feels clearer now.

The project should borrow the Logseq concept kernel, but not inherit the full weight of the Logseq product.

That distinction matters because the wiki I want is not mainly an editor. It is not a block-writing app first. It is a history-first system that slowly turns my own records into pages, links, and timelines I can revisit later.

1. What is worth borrowing from Logseq

The strongest thing in Logseq is not one feature. It is the shape of the model.

  • Page — stable things deserve stable surfaces: people, places, projects, tools, topics.
  • Block — not every meaningful unit is a full page; many things begin as fragments.
  • Link — knowledge becomes usable when one page can naturally point to another.
  • Backlink — a thing should also show where else it appears.
  • Journal / time structure — history matters, and notes often begin in time before they become categories.
  • Graph readiness — even if the UI stays simple, the data underneath should be able to describe relationships.

That is the part worth keeping.

It gives the wiki a strong inner skeleton without forcing it to become a second note-taking app.

2. What should not be copied whole

The project does not need to absorb everything around Logseq.

It does not need to start with a full editor, plugin ecosystem, whiteboard, advanced local app runtime, or all the complexity that comes with supporting a general-purpose PKM tool.

The wiki has a different center of gravity.

The hard problem here is not editing blocks beautifully. The hard problem is turning messy personal history into something structured, safe, and browseable.

That means the real custom work belongs elsewhere.

3. The actual kernel this project needs

The right framing now is not “use Logseq” or “rebuild Logseq.”

It is this:

Build a Logseq-inspired knowledge kernel.

In practice, that means the wiki should have:

  • pages for stable entities
  • smaller fact-like or mention-like blocks underneath
  • typed relations between things
  • timeline views for history-first browsing
  • backlinks and related links as default navigation
  • source evidence for where extracted information came from

This keeps the project small enough to control, but structured enough to grow.

3.1 A small self-introduction inside the architecture

I am not trying to build software for everyone here.

I am trying to build a place where my own projects, people, tools, writing, and history can sit near each other without disappearing into folders or feeds.

That is why the system should feel closer to a quiet personal encyclopedia than a productivity cockpit.

4. The six-layer architecture

The wiki now makes more sense as six simple layers.

Layer 1 — Source layer

This is where raw material comes from.

Right now the strongest source is HISTORY.md. Later it can include notes, exports, issue data, custom files, and other structured records.

Layer 2 — Extraction layer

This is the layer Logseq does not solve for this project.

History is not already a wiki. It has to be split, recognized, normalized, and filtered.

This layer should extract things like:

  • people
  • projects
  • places
  • tools
  • events
  • preferences
  • state changes
  • evidence spans

For now, rule-based extraction already exists in the repo. Later, this can become LLM + rules.

Layer 3 — Canonical data layer

Once extracted, the information needs a home that is local, queryable, and safe.

That is why the local SQLite layer matters. It gives the project a real working memory without pushing private raw history to GitHub.

Layer 4 — Materialization layer

This layer turns structured data into actual wiki pages.

That includes generated entries, page sections, related links, timelines, and search artifacts.

Layer 5 — Front-end layer

This is the visible shell: static HTML, minimal styling, and fast browsing.

The current direction still feels right: minimalist, Wikipedia-like, and calm.

Layer 6 — Review and safety layer

This is the difference between an interesting experiment and a reliable personal system.

Extraction will make mistakes. Names will need merging. Some things should stay local. Some facts will need confirmation before publication.

Without this layer, the wiki becomes noisy very quickly.

5. What the repo already has

The useful thing is that this architecture is not theoretical anymore. Parts of it already exist.

  • static HTML entry pages
  • Wikipedia-like front-end layout
  • bilingual UI framework
  • search index generation
  • rule-based extraction from HISTORY.md
  • local SQLite database path and build step
  • early generated-entry pipeline pieces

So the project is no longer at “idea only.” It has foundations. What it needs now is a cleaner framing and a more explicit MVP map.

6. Reframing the MVPs

The old issue list was mostly labeled MVP4 because the later intelligent extraction work was the clearest unresolved next step.

But looking at the repo now, the better framing is:

MVP1 — Static wiki foundation

  • clean static pages
  • search
  • minimal style
  • basic related / backlink feel

MVP2 — Better structure and browse

  • more consistent entry templates
  • stronger metadata
  • better timeline-oriented navigation
  • clearer page sections and generated page structure

MVP3 — Real personal data pipeline

  • history extraction
  • local database
  • privacy-aware generation
  • generated pages from real source material

MVP4 — Smarter extraction and review

  • better inclusion boundaries
  • LLM-assisted extraction
  • confidence and validation
  • review workflow before publishing

This sequence matches the current progress much better.

7. What is still missing even after borrowing Logseq

This is the important reality check.

Even if the project borrows page, block, backlink, and journal concepts from Logseq, it still needs the custom layer that does the real transformation.

That missing layer is the extraction pipeline.

Without it, the wiki is only a static shell.

With it, the wiki can slowly become a real personal knowledge surface.

8. The practical direction from here

So the next move is not to chase a bigger framework.

The next move is to sharpen the small framework that already exists:

  • state the architecture clearly
  • update the repo README to match reality
  • reshape the wiki GitHub issues so MVP1, MVP2, and MVP3 reflect the work already done and the work still missing
  • keep MVP4 focused on smarter extraction only after the earlier layers are stable

That feels like the right scale.

Quiet. Clear. Real.

9. The line I want to keep

If I reduce the whole direction to one sentence, it is this:

Do not rebuild Logseq. Borrow its inner grammar, then use that grammar to turn history into a calm personal wiki.
