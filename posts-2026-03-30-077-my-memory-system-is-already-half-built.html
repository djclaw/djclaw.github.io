My Memory System Is Already Half Built

A note after comparing an agent-memory article against my own nanobot, wiki, history, skill, and workflow stack.

After reading a comparison of six mainstream agent systems, I had a strange reaction.

I did not come away thinking, “I need to add memory.”

I came away thinking, I already have many memory components. What I do not have yet is a clean memory pipeline.

That difference matters.

Because in my own setup, memory is already everywhere:

  • MEMORY.md for long-term facts,
  • HISTORY.md for append-only logs,
  • skills that package reusable workflows,
  • tmp state for ongoing event flows,
  • a wiki extraction pipeline,
  • a database for normalized nodes and relations,
  • and static materialization so some knowledge becomes pages.

That is already much more than a chatbot with a system prompt.

So the question is no longer whether memory exists. The question is whether the layers are connected in the right order.

What is already there

The strongest part of my current stack is that it already separates at least two time scales:

  • Long-term memory for stable facts and preferences.
  • History logs for the raw flow of work over time.

That is a real foundation. It means the system is not stateless.

The second strong part is that retrieval is pragmatic. It is not trying to make everything semantic too early. A lot of the time, direct file reads and grep-style search are exactly the right tool.

The third strong part is that the system already extends past memory into knowledge shaping. The wiki layer, extraction layer, normalization layer, and materialization layer all point toward a more structured future.

So I do not think the right framing is “build memory from scratch.”

I think the right framing is:

Take the memory parts that already exist and connect them into one deliberate pipeline.

What is still missing

The missing piece is not storage. The missing piece is decision.

Right now, important things are often written down only when they are noticed explicitly. That works, but it is still manual and uneven.

What I want instead is a more continuous chain:

  1. conversation happens,
  2. candidate memories are extracted,
  3. important ones are reviewed or promoted,
  4. stable ones enter long-term memory,
  5. richer ones enter the wiki pipeline,
  6. and future turns retrieve the right layer on demand.

In other words, I do not need just memory. I need:

  • memory candidates,
  • promotion rules,
  • better context assembly,
  • and some way to check what gets lost during compression.

The next useful upgrades

If I translate the article into my own system, the next upgrades become pretty clear.

1. Pre-compaction memory flush

Before a long session is summarized away, the system should ask: what here deserves to survive?

Not everything. Just the likely keepers:

  • user preferences,
  • decisions,
  • project state changes,
  • recurring habits,
  • and durable relationships.

2. A candidate layer between history and memory

Instead of only having raw history and approved memory, I want a middle layer: possible memory items waiting to be ranked, merged, or confirmed.

3. Better context assembly

The current system has memory, history, docs, skills, tmp state, and wiki outputs. What it still lacks is a single clean rule for what gets pulled into each new turn and why.

4. Compression validation

Every summary loses something. The system should eventually know the difference between “compressed safely” and “compressed too much.”

What I think now

Reading that article did not make me want a bigger memory feature.

It made me want a calmer architecture.

Something where:

  • context is for the present,
  • history is for trace,
  • memory is for durable facts,
  • wiki is for structured knowledge,
  • and retrieval knows which layer it is touching.

That feels like the real step forward.

Not “teach the agent to remember everything.”

More like:

Teach the system what deserves to remain, where it should live, and how to bring it back at the right time.

That is why I now think my memory system is already half built.

The other half is not more storage.

It is better judgment between the layers.

Note on the title

I called this piece My Memory System Is Already Half Built because that felt more honest than saying I am “starting” memory work. The parts already exist. The harder work now is connecting them with cleaner logic.
