给 Telegram 加一个 /sub 轻入口

不是每次都要写“帮我开一个 subagent 做这个”。有时候，更顺手的办法，是先点一下 /sub，再把任务直接发出去。

这次做 /sub，其实和前面的 /event 是同一条思路。

不是把所有逻辑都塞进 Telegram 命令本身，而是把命令变成一个很轻的入口。

点一下，进入上下文。再发一条正常消息，让系统接住它。

为什么要做这个

原来的 subagent 调用当然能用。

但如果每次都要显式说“帮我开个 subagent 去做这个任务”，交互会有一点重。

尤其在 Telegram 里，这种重量会更明显。

手机上最自然的方式，通常不是先组织一条完整命令，而是先进入某个模式，再把自然语言内容补上。

所以这次的目标很简单：

  • /sub 只负责进入 sub mode
  • 下一条普通消息，默认当成要委派给后台 subagent 的任务
  • 如果用户说 cancel / 取消，就退出这个模式

它现在怎么工作

  1. 用户在 Telegram 里点 /sub
  2. bot 返回一小段 intro，并进入 sub mode
  3. 用户发送下一条普通自然语言消息
  4. 系统把这条消息当成 subagent 的任务 payload
  5. 然后调用后台 subagent，主线程只返回一个简短确认

现在的 intro 很短：

Subagent is ready.
Send me the task you want delegated.
I will treat your next normal message as a background sub-task.
Say cancel to leave sub mode.

重点不是花哨，而是低摩擦。

让用户不用重新学习一套新的命令语言。

实现上怎么拆

这次我还是沿用了一个我现在越来越喜欢的结构：

  • 安装包里只放薄补丁
  • 真正的业务逻辑放在 workspace
  • skill 层只负责说明 workflow 和边界

这么做的原因很现实。

nanobot 升级以后，安装包目录里的改动最容易被覆盖。

如果把主要逻辑都写在 site-packages 里面，每次升级之后都要重新找、重新补、重新对齐，非常麻烦。

所以这次故意把核心逻辑放回 workspace，只在安装包里留最薄的一层接线。

这次新增了哪些文件

workspace 里主要新增了三个文件：

  • scripts/telegram_sub.py
  • scripts/telegram_sub_cli.py
  • skills/sub/SKILL.md

它们各自做的事情也很清楚：

  • telegram_sub.py：保存 sub mode session、处理 enter / handle / cancel、把自然语言转成可委派任务
  • telegram_sub_cli.py：做成一个很薄的 CLI 入口，方便安装包里的 loop 调它
  • skills/sub/SKILL.md：定义这个 workflow 是什么、适合什么场景、交互应该有多轻

安装包里改了什么

安装包里只改了两块：

  • nanobot/channels/telegram.py
  • nanobot/agent/loop.py

前者负责 Telegram 菜单注册和命令入口。

后者负责在运行时识别 /sub，以及在 sub mode 下把下一条普通消息转给 workspace 里的脚本处理。

也就是说，安装包做的是“接线”。

真正的行为定义，还是在 workspace。

为什么还要有一个 skill

很多人会问，既然已经有脚本了，为什么还要保留一个 skills/sub/SKILL.md？

因为脚本负责跑，skill 负责讲清楚这个东西是什么。

它们不是重复，而是两层不同的东西。

  • 脚本：负责 session、命令解析、状态切换、CLI 输出
  • skill：负责目标、交互原则、适用范围、质量边界

有了这个 skill，后面如果继续扩展 /sub，就不会只剩下一堆零散代码。

它会有一份明确的 workflow 说明，知道自己应该轻到什么程度，不该膨胀成什么样。

升级 nanobot 之后怎么办

这一点反而是这篇里最值得记下来的。

因为 nanobot 一升级，site-packages 里的补丁很可能就没了。

所以这套结构的维护方法应该是：

  1. 把主要逻辑保留在 workspace
  2. 把安装包里的改动压到最薄
  3. 升级后只需要重新补回 Telegram 菜单和 loop 接线
  4. skill 文件和 workspace 脚本本身不用重写

换句话说，升级后的恢复重点不是“把整个功能重做一遍”。

而是“把那一层薄补丁重新打上去”。

这也是为什么我现在更偏向这种结构：

installed package 是 adapter，workspace 才是 source of truth。

当前状态

/sub 现在已经具备这些能力：

  • Telegram 菜单里可注册为独立命令
  • 点 /sub 后进入 sub mode
  • 下一条普通消息可直接作为 subagent 任务
  • 支持 cancel / 取消 退出
  • workspace 逻辑和安装包接线已经拆开

它很小，但方向是对的。

比起再发明一条更复杂的命令，它更像是在 Telegram 里给 agent 留了一个顺手的小入口。
