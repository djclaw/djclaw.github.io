From HISTORY.md to a Living Wiki

Not one magical prompt. A small pipeline: sources, extraction, normalization, review, storage, then materialization. And right now, the PoC only covers part of that path.

When people say “just use AI to turn your notes into a wiki,” it sounds like one step.

But the more I work on this personal wiki, the more it feels like the opposite. A good wiki is not one giant model call. It is a chain of smaller layers, each doing one quieter job.

This week’s extraction PoC made that clearer.

1. The six layers

The shape I want now is simple:

  1. Sources
  2. Structured extraction
  3. Normalization
  4. Review & safety
  5. Canonical storage
  6. Materialization

That is the path from a raw line in HISTORY.md to an actual wiki page that can be browsed, linked, and trusted.

2. Layer 1: Sources

This layer is the intake desk. Its job is not to be smart. Its job is to make raw inputs look consistent enough for the next layer.

Right now the PoC only uses one source: memory/HISTORY.md. It reads timestamped lines, keeps the original text, and writes a recent sample set to wiki/data/extraction-poc/recent-history-sample.json.

A tiny example:

{
  "source_id": "history_md",
  "timestamp": "2026-03-25T19:16",
  "raw_text": "[2026-03-25 19:16] User pivoted the wiki project toward a Logseq-inspired architecture..."
}

That is already a source layer, just a very narrow one. Later, this same layer should also handle notes, markdown docs, project files, issue logs, and exports.

3. Layer 2: Structured extraction

This is the first layer that feels like “AI.” It takes a source chunk and asks:

  • What entities are here?
  • What events happened?
  • What relations are worth keeping?

In the PoC, this is the strongest implemented layer so far. The script asks the model for JSON with nodes, relations, and events, and keeps the original evidence next to the extracted candidates.

For example, a line about the trips project might produce:

  • trips as a project
  • New Orleans as a place
  • Airbnb as an org or reference point
  • an event like “trips repo approved”

This part is real now. It is not just summarization. It is typed candidate extraction.

4. Layer 3: Normalization

This is the layer that stops the wiki from becoming a mess of duplicates.

Extraction alone is not enough. Models will happily give you:

  • djclaw
  • DJClaw site
  • djclaw.github.io

Those should probably become one stable thing: DJClaw.

The current PoC already does a small handcrafted pass here. It normalizes some labels, filters junk, fixes a few types, and deduplicates repeated candidates.

So if the model returns github pages, the current code can map it toward GitHub Pages. If it returns a filename or a hash, the current code can often drop it before it pollutes the graph.

But this layer is still small. It is embedded inside the PoC parser, not yet a full subsystem with alias dictionaries, merge suggestions, or cross-entry canonicalization.

5. Layer 4: Review & safety

This is the layer that reminds me not to trust a pretty output too quickly.

A wiki that grows from personal logs has to care about things like:

  • privacy
  • sensitivity
  • publishable vs local-only
  • merge conflicts
  • low-confidence guesses

Right now, this is mostly not implemented as a real workflow. The PoC is reviewable, because it preserves raw evidence, raw model output, timestamps, and parse errors. But there is no review queue yet, no approval UI, and no proper privacy classification.

So this layer exists more as a principle than as a finished feature.

6. Layer 5: Canonical storage

This is where reviewed knowledge becomes stable local state.

In the broader wiki repo, this already exists as a direction. The local kernel database now uses tables like:

  • sources
  • evidence
  • nodes
  • relations
  • events

But the extraction PoC does not write into that canonical database yet. It still stops at local candidate files such as structured-candidates.json.

That means the storage layer exists in the project, but the new extraction experiment is not connected to it yet.

7. Layer 6: Materialization

This is the part people usually imagine first: pages, search, timelines, related links.

But materialization should come last.

If extraction is noisy and normalization is weak, then materialization just turns mistakes into polished mistakes.

The broader wiki already has page-generation and search-index ideas. But this PoC still does not feed that layer. It can tell me a HISTORY entry probably mentions Logseq or DJClaw; it cannot yet automatically regenerate a trustworthy wiki page from those candidates.

8. What the current PoC actually covers

If I zoom out, the current experiment is strongest in the middle:

  • Layer 1: lightly implemented for recent HISTORY.md only
  • Layer 2: yes, this is the core of the PoC
  • Layer 3: partially implemented as local normalization rules
  • Layer 4: reviewable, but not a real review system
  • Layer 5: not connected yet
  • Layer 6: not connected yet

So the honest summary is:

The PoC proves that typed extraction from recent personal history is possible, and that a small normalization pass already helps. But it is still pre-review, pre-storage-integration, and pre-materialization.

9. A few concrete examples

Here are a few simple examples of what these layers really mean in practice.

Example A: one label, many variants

Suppose the source text mentions:

  • djclaw
  • DJClaw site
  • djclaw.github.io

Extraction may see three things. Normalization should say: no, this is one canonical node.

Example B: useful event vs pretty sentence

Given a line about publishing post 067, a weak system might produce only a nice summary paragraph. A stronger extraction layer should instead produce:

  • an event candidate
  • the relevant node candidates such as Logseq and DJClaw
  • evidence tied to the original log line

Example C: public wiki vs local knowledge

If a source line contains a private path, a secret-like token, or sensitive personal detail, extraction may still notice it. Review & safety should decide it does not become a public page.

10. Why this layered view matters

The main reason is simple: it stops me from pretending that extraction alone is the system.

One model call can be useful. But a living wiki needs more than “the model said so.” It needs evidence, normalization, review, stable storage, and only then a page.

That is slower than the magical version.

But it is also how the project stays local-first, debuggable, and worth trusting later.

11. Where I would go next

If this path keeps moving, the next steps feel clear:

  1. keep improving extraction quality on a small sample
  2. pull normalization into a cleaner module
  3. add explicit review and privacy handling
  4. only then connect candidate output into canonical storage
  5. and only after that let it materialize pages more directly

So for now, this is where we are:

not one magical prompt, but a small workshop.

Messy logs in. Quiet pages out. A few careful layers in between.
