Skills as a Product Pipeline

Why Matt Pocock’s skills repo feels less like a prompt library and more like a small operating system for building products.

I spent time reading Matt Pocock’s skills repo. The interesting thing is not that it contains many skills. A lot of repositories contain many things. The interesting thing is that the repo quietly implies a way of working.

It does not feel like a bag of prompts. It feels like a conveyor belt.

What the repo actually is

At first glance, the repo is simple: one folder per skill, a SKILL.md in each one, and a root README that groups them by job to be done. But after a while, the pattern becomes clearer.

These skills are not trying to be clever in isolation. They are trying to make work routable, repeatable, and durable.

That means three things matter a lot:

  • the trigger has to be clear,
  • the output has to become an artifact,
  • and one skill should be able to hand off to the next.

What it gets right

The best skills in the repo are small, opinionated, and concrete. They say what they do. They say when to use them. And they end in something you can keep using after the chat window is gone.

That is a big difference from many "AI workflow" collections. A lot of those are really just reusable moods. This repo is closer to reusable operating procedure.

I kept seeing the same pattern:

  • write-a-prd gives you a product document,
  • prd-to-plan turns it into phases,
  • prd-to-issues turns it into work units,
  • tdd gives implementation discipline,
  • triage-issue feeds mistakes back into the loop,
  • edit-article helps package the result.

That is not a list of disconnected helpers. That is a product pipeline hiding in plain sight.

The implied pipeline

Once I stopped reading the repo as a catalog and started reading it as a sequence, the shape became obvious:

  1. clarify the problem,
  2. write the PRD,
  3. explore interface or architecture boundaries,
  4. turn the idea into a phased plan,
  5. turn the plan into issues,
  6. implement slice by slice,
  7. triage what breaks,
  8. publish what was learned.

That is already enough to build a lightweight product development system.

Why this matters more than better prompts

I think the real lesson here is simple: the value is not in saying smarter things to a model. The value is in creating clean handoffs.

If every step ends in a durable artifact, then the next step has a place to stand. If every skill has a clear trigger, then the system can route work with less confusion. If the outputs are structured, then the process starts to compound.

That is much more important than whether a skill feels "powerful" in one chat.

What I would borrow

If I were turning this style into my own product development pipeline, I would keep a canonical chain like this:

  • problem brief
  • PRD
  • architecture / interface memo
  • implementation plan
  • issue breakdown
  • execution loop
  • bug triage
  • retro / release note

And for every stage, I would require the same questions:

  • When should this be used?
  • What does it take in?
  • What artifact does it emit?
  • Does it need human approval?
  • What system of record should it update?

What I would add

The repo is strongest on software and coding flow. If I were extending it, I would add a few business-facing layers too:

  • customer research synthesis,
  • metrics and success-criteria definition,
  • launch checklist,
  • retrospective / postmortem,
  • release notes and changelog packaging.

That would make the conveyor belt feel more complete from idea to shipped learning.

What I am taking from it

The quiet insight I am keeping is this: a good skill is not just a reusable prompt. It is a reusable handoff.

That means a skill should not only help think. It should help move work from one shape into the next shape without losing clarity on the way.

That is why this repo stayed with me. It is not trying to be magical. It is trying to make product work legible.

And that is exactly the kind of thing I want to build on DJClaw too.
