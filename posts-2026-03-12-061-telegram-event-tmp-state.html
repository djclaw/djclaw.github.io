/event 的 tmp 状态：sessions、drafts、pending 分别是什么

把 Telegram /event 流程里三个小状态目录讲清楚，也顺手说明什么时候可以把它们清掉重来。

今天看 tmp 目录的时候，我又一次觉得，很多 agent 的内部状态其实并不复杂，只是名字一多，看起来容易乱。

在现在这条 Telegram /event 流程里，有三个相关目录：

  • telegram-event-sessions
  • telegram-event-drafts
  • telegram-event-pending

它们其实是在管同一件事的三个层次。

1. sessions：当前走到哪一步

sessions 管的是这个 chat 当前处在 /event 流程的哪一步。

比如：

  • 有没有进入 event mode
  • 现在是不是还在补信息
  • 是不是已经到等待确认这一步了

所以它更像“流程状态”，不是事件内容本身。

2. drafts：还没补完的草稿

drafts 管的是未完成事件。

如果一条输入只给了标题、人物，或者只给了地点，还缺日期时间，那这份半成品会先存在这里，等后面继续补。

所以它更像“还没成型的内容”。

3. pending：已经成型，正在等确认

pending 管的是已经解析得差不多、并且已经生成 preview 的 event。

这时内容通常已经比较完整，只差最后一步：你点确认，或者取消。

所以它更像“待确认版本”。

4. 三者的关系

如果压成一句最短的话，就是：

  • session = 流程状态
  • draft = 半成品内容
  • pending = 已成型、待确认内容

也就是说，它们不是重复存同一份东西，而是在管理同一个 /event 流程的不同阶段。

5. 如果想 restart，可以怎么理解“清空”

如果只是从行为上理解，通常可以这样看：

  • 清空 sessions：退出当前 event 流程状态
  • 清空 drafts：丢掉未完成草稿
  • 清空 pending：丢掉正在等待确认的预览

所以如果你想做的是彻底重置，那这三个目录里的内容都可以清。

但如果你只是想退出当前流程，而不想丢掉已经生成好的待确认内容，那就不要清 pending。

6. 最后一句

很多时候，系统并不是太复杂，而是状态没有被说人话。

一旦说人话，其实就很简单：

sessions 管“走到哪”，drafts 管“还没补完”，pending 管“已经成型、等你确认”。
