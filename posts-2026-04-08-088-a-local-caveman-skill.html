A Local Caveman Skill for Bilingual Conversations

A small note on studying an existing compression-style skill, then rebuilding it in a way that fits my own agent stack better.

I spent a little time looking at JuliusBrussee/caveman and its docs. The idea is simple and good: if an assistant can say the same thing in fewer tokens, it should. Less filler. Less ceremony. More signal.

I liked the idea immediately. But I also knew I probably did not want to use it exactly as-is.

The original project is shaped for Claude Code / Codex-style environments. It makes a lot of sense there. But my actual daily workflow is now a little different. I talk to nanobot inside Telegram and through my own workspace. I also switch between English and Chinese a lot. That changes the right design.

What the original caveman idea gets right

The core instinct is correct: most assistant answers are longer than they need to be. A good compression layer can make responses faster, cheaper, and easier to scan. This is especially useful for technical work, short planning loops, and back-and-forth execution.

But there is a difference between short and damaged. If a style gets too compressed, it can start to lose trust, precision, or even basic readability.

Why I made a local version instead

For my own stack, the upstream plugin was not the real target. The real target was this:

  • work inside my existing nanobot workspace,
  • support both English and Chinese,
  • stay concise without sounding broken,
  • protect commands, file paths, warnings, and other details that should never be compressed too aggressively.

So instead of just installing the official version and hoping it would fit, I made a local caveman skill in my own workspace.

The bilingual part matters more than it looks

English can tolerate a more obvious “caveman” style. You can drop filler, softeners, and a lot of connective phrasing, and the result still feels readable.

Chinese is different. If you compress too literally, it starts to sound less like “efficient” and more like “broken translation.” So the Chinese side of this skill is not really trying to imitate caveman speech. It is trying to become clean, direct, and light.

That led to a better split:

  • English: allow more obvious compression.
  • Chinese: keep natural grammar, but remove fluff and lead with the answer.

Three levels were enough

I kept the local version simple:

  • lite — concise but still normal,
  • full — noticeably tighter,
  • ultra — maximum compression without breaking safety.

That feels like the right amount of control. Not a giant style system. Just three useful levels.

What should never be compressed too far

This turned out to be the most important part of the local rewrite.

Some things should stay exact, even when the rest of the answer gets shorter:

  • shell commands,
  • file paths,
  • code,
  • dates and times,
  • approval boundaries,
  • medical / legal / financial caution,
  • destructive-operation warnings.

So the real rule became: compress style, not truth.

Current state

The local skill exists now as a workspace skill spec. It is not yet a fully wired runtime mode with a Telegram command and persistent per-chat state. That can come next.

But even this first step already matters. It turns a vague taste preference — “please be shorter” — into something more explicit and reusable.

Why this feels useful

I keep coming back to the same small lesson: many good agent features do not begin as big systems. They begin as a tiny rule about behavior, phrased clearly enough that it can be reused.

This one is just a compression rule. But it is a useful one. And because it now lives in the workspace, it can evolve with the rest of the stack.
