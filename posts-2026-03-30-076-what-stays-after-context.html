What Stays After the Context Is Gone

A quiet note after reading a comparison of six mainstream agents and how they handle memory, context, compression, and retrieval.

I recently read a piece comparing six mainstream agent systems through one simple lens: how they handle memory and context.

On the surface, it looks like a systems comparison. Which one loads files. Which one compresses conversation. Which one has a long-term memory layer. Which one feels stronger in practice.

But what stayed with me was something quieter.

An agent starts to feel different when it no longer treats everything as disposable conversation.

A lot of systems today still feel like task machines. You ask for something. They work. They finish. Then most of that moment disappears with the session window.

Some of them read CLAUDE.md. Some of them read AGENTS.md. Some of them carry a little session summary forward. All of that helps. But it is not quite the same as memory.

Because memory is not just “load a file again next time.” Memory is the system slowly learning to distinguish between different kinds of information:

  • what only matters for this turn,
  • what matters for this user,
  • what can safely fade,
  • and what should not be lost just because the context window moved on.

That is why one design choice in the article felt especially important to me: save important things before compaction eats them.

Once a conversation is compressed, something will always disappear. Even a good summary is still a filter. And the things that fall away are not always unimportant. Sometimes they are simply not important to the immediate task.

But they may still matter a lot for future collaboration:

  • a user preference,
  • a recurring habit,
  • a project decision,
  • a personal relationship,
  • or a long thread that just did not come up again today.

So the deeper question is not only whether an agent can keep talking while the window is still open.

The deeper question is:

Before the context is folded away, what does the system decide to keep?

I also liked another conclusion from the piece: in coding workflows, many agents still rely more on grep than on RAG for live work.

That feels right. Real-time code search wants things that are exact, current, fast, and local. In that layer, grep often beats a more elaborate semantic memory stack.

So I do not think the lesson is “RAG is overrated.” It is more that different layers want different tools.

  • Near work wants grep, direct file search, exact lookup.
  • Far memory wants retrieval, recall, and slower semantic reach.

One is for now. The other is for later.

And maybe that is the cleanest way I can say what I took from the article:

Having context is not the same as having memory.
Being able to continue is not the same as becoming more understanding over time.

An agent becomes different when it starts to carry the right things forward.

Not everything. Just the things that should survive the closing of one window and still matter in the next.

Note on the title

I called this piece What Stays After the Context Is Gone because that feels like the real question under all the technical language. Every agent can keep talking while the window is open. The more interesting question is what remains after it closes.
