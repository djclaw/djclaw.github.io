034 Linko Universal Plugin-Kit v1

2026-03-04

Linko Universal Plugin-Kit v1 实施计划（基于现有 MCP，CLI 不接管 Auth）

Summary

在不改动现有 services/mcp/linko_chatgpt_app 认证与工具链路的前提下，新增一套“平台无关”的 skills + roles + 安装 CLI，放在 services/plugins/linko/。
v1 目标是：可安装、可复用、可扩展到 Codex / Claude / OpenClaw / Nanobot / Antigravity（先文档映射，不做原生适配）。
v1.1 再做 npx 分发与 skill-from-masters 扩展生成流程。

1) 目标与范围

In Scope (v1)

  1. 建立通用插件资产目录：services/plugins/linko/。
  2. 提供 4 个 role + 4 个 skill（围绕现有 3 个 MCP tools）。
  3. 提供轻量 CLI（仅 install/list/doctor）。
  4. 提供多平台目录映射文档（copy/sync 到不同生态目录）。
  5. 提供最小发布文档与版本规范（GitHub 分发）。

Out of Scope (v1)

  1. CLI 不承载 MCP 业务调用。
  2. CLI 不实现 OAuth 登录、PKCE、token 刷新。
  3. 不做 OpenClaw/Nanobot/Antigravity 原生插件格式适配器。
  4. 不改动现有 MCP/OAuth 后端实现。

2) 当前基线（必须保持）

现有 MCP 服务继续作为唯一认证与调用入口：

  1. OAuth metadata + DCR + authorize + token + revoke：由 services/mcp/linko_chatgpt_app/oauth_server.py 提供。
  2. MCP token 校验：由 services/mcp/linko_chatgpt_app/token_verifier.py 调 Linko /api/user/me/。
  3. Tool 逻辑：get_notes / get_resources / get_subjects 在 services/mcp/linko_chatgpt_app/mcp_server.py。
  4. CLI 与 skills 只“消费”该服务，不替代该服务。

4) 角色与技能定义（v1）

4.1 Roles（4个）

  1. linko-orchestrator：任务拆解、工具选择、结果编排。
  2. linko-study-planner：学习目标拆分、主题路径建议。
  3. linko-resource-curator：资源筛选、按 subject 聚合。
  4. linko-content-writer：基于笔记/资源生成结构化内容。

4.2 Skills（4个）

  1. linko-orchestration：何时调用哪个 tool，失败回退策略。
  2. linko-study-planning：以 subject 为核心的规划流程。
  3. linko-resource-curation：资源召回、过滤与排序建议。
  4. linko-content-writing：输出模板与证据引用规则。

4.3 每个 Skill 必含契约

  1. 触发条件（when to use）。
  2. 输入参数规范（含缺失参数补问策略）。
  3. MCP tool 映射（调用 get_notes|get_resources|get_subjects 的条件）。
  4. 错误/无数据/权限失败时的标准回复模板。
  5. 输出格式（summary, evidence, next_action）。

5) CLI 设计（v1）

5.1 命令

  1. linko-plugin list
  2. linko-plugin install --target <codex|claude|custom> --dest <path> --mcp-url <url>
  3. linko-plugin doctor --target <...> --dest <path> --mcp-url <url>

5.2 行为边界（强约束）

  1. 只做文件安装、目录映射、配置检查。
  2. 不执行 OAuth 流程。
  3. 不存储 access/refresh token。
  4. 不代理 MCP 请求。

5.3 target 映射策略

  1. codex`：安装到用户指定 skills 目录（或 .agents/skills` 这类自定义路径）。
  2. claude`：安装到 .claude/agents` 与对应 skills 路径。
  3. custom`：用户显式给 --dest`，CLI 仅做结构校验与复制。

6) 公共接口 / 类型新增

6.1 manifest.yaml（新增）

字段：

  1. name
  2. version
  3. mcp.base_url
  4. skills[]
  5. roles[]
  6. targets[]
  7. compatibility[]

6.2 CLI 输出约定

  1. 成功：code=0，输出安装/校验报告（JSON + 人类可读摘要）。
  2. 失败：code!=0，标准错误码：
    • E_TARGET_UNKNOWN
    • E_DEST_INVALID
    • E_MCP_UNREACHABLE
    • E_COPY_FAILED
    • E_VALIDATION_FAILED

6.3 文档契约

  1. docs/targets.md 明确每个平台需要用户手动放置的目录。
  2. docs/compatibility.md 明确“通用包支持”和“原生插件支持”区别。

7) 分阶段实施

Phase 1：骨架与契约

  1. 创建 services/plugins/linko/ 完整目录。
  2. 定义 manifest.yaml 与 CLI 错误码。
  3. 定义 4 roles/4 skills 模板骨架。

Phase 2：内容实现

  1. 完成 4 个 role md。
  2. 完成 4 个 SKILL.md（绑定现有 MCP tools）。
  3. 完成 docs 初版（安装、映射、限制说明）。

Phase 3：CLI 实现

  1. 实现 list/install/doctor。
  2. 支持 target 与 custom dest。
  3. 加入最小校验（目录权限、MCP URL 可达性、manifest 一致性）。

Phase 4：验证与发布

  1. 本地 smoke：从空目录安装到 codex/claude/custom。
  2. 文档跑通：按文档从零完成安装。
  3. GitHub release（v1）。

Phase 5：v1.1

  1. npx 分发。
  2. 引入 skill-from-masters 批量生成/扩展 skills 与 roles。
  3. 根据反馈评估是否增加原生平台 adapter。

8) 测试用例与场景

安装类

  1. install --target codex 成功复制并生成报告。
  2. install --target claude 成功复制到指定目录。
  3. install --target custom --dest ... 支持任意路径。

校验类

  1. doctor 在 MCP URL 不可达时返回 E_MCP_UNREACHABLE。
  2. manifest 缺字段时返回 E_VALIDATION_FAILED。
  3. 目标目录无写权限时返回 E_DEST_INVALID。

内容类

  1. 每个 role 至少 2 条提示语可触发正确 skill。
  2. skill 能正确指引调用三种现有工具。
  3. 无数据与认证失败时输出符合预定义模板。

回归类

  1. 不修改现有 MCP 代码路径。
  2. 现有 ChatGPT App OAuth+tools smoke 结果不受影响。

9) 风险与缓解

  1. 平台目录规范变化：使用 custom target + 文档映射避免硬编码。
  2. 用户误解“CLI=认证客户端”：在 README 强调 CLI 不处理 auth。
  3. 多生态兼容膨胀：v1 只做通用层，v1.1 再评估原生适配。

10) 验收标准（Definition of Done）

  1. services/plugins/linko/ 目录完整，含 4 roles、4 skills、CLI、docs、manifest。
  2. CLI 三个命令可执行且有稳定退出码。
  3. 任意新用户按文档可在 10 分钟内完成安装并跑通一次真实 MCP 使用流程。
  4. 不引入对现有 services/mcp/linko_chatgpt_app 的行为变化。
  5. 发布 v1 tag + 安装文档可复现。

Assumptions and Defaults

  1. 现有 MCP 服务域名、OAuth 与 Railway 部署已可用。
  2. v1 分发渠道以 GitHub 为主，不要求 npm/pypi 同步上线。
  3. 用户可自行决定目标 agent 工具目录；CLI 负责拷贝和检查，不负责工具注册生命周期。
  4. skill-from-masters 仅在 v1.1 引入，不作为 v1 阻塞项。
