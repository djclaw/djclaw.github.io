为什么 Telegram 里的 /event 会直接发出去

以及：怎么把它改成先放进输入框，不自动发送。

这个问题很小，但真的很影响手感。

原来的流程是：我在 Telegram 里点一下菜单里的 /event，它就直接发出去了。然后 bot 收到一个空的 /event，马上回我一段 usage 说明。技术上没错，但体验很别扭。

我真正想要的其实是另一种感觉：

点一下 /event，先把它放进输入框里，别急着发。我再继续补时间、地点、人物、链接，最后自己按发送。

这次修改，解决的就是这个差别。

改之前

之前的行为是：

  1. 点击 Telegram 菜单里的 /event
  2. Telegram 立刻把 /event 当成命令发出去
  3. bot 收到的是一个没有正文的 /event
  4. 于是返回提示：你还没写活动内容

结果就是，用户脑子里想的是“先开个头”，系统理解成了“现在就执行”。

改之后

现在的流程变成了：

  1. 先使用 /eventmenu
  2. bot 回一个按钮
  3. 点这个按钮后，/event 会被插入当前聊天输入框
  4. Telegram 不会自动发送
  5. 用户可以继续补完整内容，再手动发送

也就是说，现在它更像一个写作起手式，而不是立刻执行的命令。

前后对比

Before

  • 点击 /event
  • 立即发送
  • 收到 usage 提示
  • 用户还得重新写一遍

After

  • 发送 /eventmenu
  • 点击插入按钮
  • /event 进入输入框
  • 不自动发送
  • 用户把内容补完后再发

问题到底出在哪

一开始我以为是 bot 自己太急了，后来发现不是。

真正的原因在 Telegram 自己的两种交互方式不一样：

  • 一种是 bot command：点了就发
  • 另一种是输入框插入：只是把文字放进去，先不发

而原来的 /event 被注册成了标准 bot command，所以 Telegram 的默认动作就是：直接发送。

这不是 parser 的 bug，也不是 nanobot 额外做了什么奇怪动作。它只是老老实实地接住了 Telegram 送来的空命令，然后给出说明。

怎么改的

修法其实很简单，关键是把两个角色拆开。

  • /event：保留，继续作为真正的后台解析命令
  • /eventmenu：新增，专门负责更自然的手机端输入体验

也就是说：

/event 负责“执行”，/eventmenu 负责“起手”。

这样做以后，底层解析流程基本没动。变的只是前面那一下交互入口。

为什么不能直接让菜单里的 /event 只插入不发送

这里有一个很现实的限制：Telegram 的标准 slash-command 菜单，本来就不是“草稿模板系统”。它的设计更接近“命令触发器”。

所以如果要求是：

我就想点菜单里的 /event，但它不要发，只插进去。

那标准 bot command 这条路本身就不太支持。要实现“插进去但不发送”，就得换另一种 Telegram 提供的交互方式。

所以这次不是硬拗一个不自然的 hack，而是顺着 Telegram 本来的能力，把“命令”和“输入辅助”拆成了两件事。

为什么这个小改动值得做

因为它把节奏还给了人。

原来是系统在催你：你点了，那我就立刻执行。现在则变成：我先帮你把开头放好，剩下的你慢慢写，准备好了再发。

这个差别看起来很小，但在手机上尤其明显。手机输入本来就慢，任何“误触即执行”的设计，都会让人觉得不顺。相反，只要先给一个半成品输入框，整个动作就会自然很多。

现在的理解

这次我也顺手把一件事想清楚了：

不是所有 slash command 都应该直接执行。有些命令其实更像写作模板的起点。

像 /event 这种需要继续补时间、地点、人物、链接的命令，本质上就更适合先进入输入框，再由人把它写完整。

所以更准确地说，这次不是“修了一个 bug”，而是把一个命令从“执行按钮”改成了“输入起点 + 执行命令”两段式。

用起来会更顺，也更像人在和一个助手配合，而不是在被一串命令推着走。
