Telegram menu，给 nanobot 留一个顺手的入口

不是多一个功能，而是多一个更顺手的入口。菜单在那儿，人就更容易真的去用。

这次做的事情很小。

不是重新造一个 skill。不是再接一个 MCP。只是把 nanobot 在 Telegram 里常用的几个动作，放进 / 菜单里。

最后先加了三个：

  • /help
  • /focus
  • /status

为什么要做这一步

很多能力其实已经有了。真正麻烦的，不是不会做，而是每次都要重新想一句触发词。

如果 Telegram 菜单里已经放着一个入口，动作会更短。脑子的阻力也更小。

/focus 这种命令尤其适合这样做。它不是复杂对话，只是一个高频动作。

这次是怎么做的

nanobot 现有的 Telegram 通道代码里，本来就已经支持 slash command。

所以这次不是从零开始，而是在已有结构上补两个入口，再把 /help 一起更新。

主要改了两层：

  • Telegram channel：注册命令菜单，并把命令转发进去
  • Agent loop：真正定义 /status 和 /focus 的行为

三个命令现在分别做什么

/help：返回可用命令列表。

/status：返回当前模型、channel、chat id、session。

/focus：先走最轻的一版，不直接写死逻辑，而是转成一句固定 prompt：让 nanobot 返回当前完整 Focus 列表。

升级之后怎么办

这次改的是已安装包里的代码。

所以以后如果 nanobot 升级、重装，补丁大概率会被覆盖。

解决办法也很简单：不要再手改一遍，把动作收进一个脚本。

这次已经把脚本放到 workspace 里了：

/home/dj/.nanobot/workspace/scripts/apply_nanobot_telegram_commands.sh

以后升级后，只要再跑一次：

/home/dj/.nanobot/workspace/scripts/apply_nanobot_telegram_commands.sh <systemd-service-name>

比如：

/home/dj/.nanobot/workspace/scripts/apply_nanobot_telegram_commands.sh my-nanobot-service

脚本里做了什么

  • 给 Telegram 菜单加上 /focus 和 /status
  • 给 Telegram handler 注册这两个命令
  • 给 agent loop 加上实际处理逻辑
  • 做一次 Python 语法检查
  • 最后尝试重启 systemd service

现在还差最后一小步

脚本本身已经写好了，补丁也已经打进去了。

但这台 Pi 上默认猜的服务名 nanobot 不存在，所以还需要确认真实的 systemd service name，再重启一次，命令菜单才会正式生效。

可以这样查：

systemctl list-units --type=service --all | grep -i bot
systemctl list-unit-files | grep -i bot

这类改动的感觉

有时候系统不是缺能力。

只是缺一个顺手的入口。

入口做对了，同样的能力才会更像日常工具，而不是一堆只在脑子里存在的功能。
