034 安静的总包装：MCP + Skills + 多角色 Agent

2026-03-04

目标很简单：把 MCP 做成核心，再把 skills 和多角色 agent 有序地包起来，最终变成别人能装、能用、能扩展的整体。

结论（先说重点）

  • 核心用 MCP + skills。这是最通用、最能在 nanobot / codex / CI 里跑的组合。
  • 插件只是包装。它是平台锁定的分发形式，不是核心能力。
  • 推荐混合分发。一个 Python 核心包 + 一个可选 skills 包，未来再加插件外壳。

为什么不从 plugin 开始

  • plugin 属于平台语法（OpenAI / Claude Code 各自一套）。
  • 在 nanobot / codex / CI 里不能直接执行。
  • 所以：先做可运行的核心，再做分发包装。

推荐结构（骨架）

repo/
  pyproject.toml
  README.md
  src/
    auto_company/
      cli.py
      mcp/
        server.py
      plugin/
        wrapper.py
      config/
        default.yaml
  skills/
    <skill_name>/
      SKILL.md
      scripts/
      assets/

CLI 入口（一个命令）

  • MCP：auto-company mcp --config ~/.auto-company/config.yaml
  • 插件包装：auto-company plugin --stdio

给 Codex 的任务清单

  1. 生成 repo 基础结构
  2. 写 CLI 入口（mcp / plugin 子命令）
  3. 搭 MCP server 基础框架
  4. 做一个最小可运行 skill
  5. 补 README（安装 / 运行 / skills 路径）
  6. 可选：写 plugin wrapper

一句话总结

先把核心能力做成 MCP + skills，再用插件当外壳，这样既能跑、又能扩。

English Version

The goal is simple: keep MCP as the core, wrap skills and multi‑role agents on top, and ship a package others can install and run.

Key decisions

  • MCP + skills are the core. They run across nanobot / codex / CI.
  • Plugins are packaging. Platform‑locked wrappers, not the core logic.
  • Ship a hybrid. One Python core package + one optional skills bundle.

Why not start from a plugin

  • Plugins are platform‑specific (OpenAI vs Claude Code).
  • They don’t run directly in nanobot / codex / CI.
  • So build the core first, wrap later.

Task list for Codex
