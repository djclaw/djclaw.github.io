043 如果只做第一周：我会怎么配置 gws agent

2026-03-08

前两篇已经把方向说清了：先看懂 gws 的 skills 结构，再挑三条最值得先跑通的链路。但真要开始试，不该一上来就追求“全自动办公”。第一周更像是在做一件小事：证明 agent 能在受控范围内，稳定完成少量真实工作。

如果是我来配第一周，我会坚持四个原则：

  • 先读后写：先让 agent 会看，再让它会改。
  • 先建议后执行：先输出建议，再允许少量落地动作。
  • 先单步后多步：先证明一个动作稳，再串工作流。
  • 先低风险后高风险：不碰删除、不碰自动回复、不碰批量改权限。

1) 第一天：只把环境跑通，不做自动化

第一天不要急着让 agent 真干活。目标只有两个：

  1. gws 能正常安装和认证；
  2. agent 能稳定调用它，并拿到结构化输出。

这一天我只做这些检查：

  • gws --help 正常；
  • 认证能完成；
  • 至少一个服务能返回 JSON；
  • agent 能读取命令输出，不发生格式误判；
  • 失败时能看懂错误信息，而不是直接胡猜。

这一步很无聊，但它其实决定了后面会不会一直返工。

2) 第二天：只开放“读”能力

第二天我会让 agent 获得这几类只读能力：

  • 读 Gmail 列表；
  • 读今日或本周 Calendar agenda；
  • 读 Tasks 列表；
  • 查联系人；
  • 列出 Drive 文件。

注意，这里我不会先问“它能不能发邮件”，而会先问：

它能不能稳定地看懂我现在的 Workspace 里发生了什么？

如果连读都不稳，写入只会把问题放大。

第二天的验收标准

  • 我说“看看我今天的日程”，它能返回 agenda；
  • 我说“找最近未读邮件”，它能列出结果；
  • 我说“查一下某个联系人”，它能找到信息；
  • 结果不是自由发挥，而是基于真实返回。

3) 第三天：只允许一类写入 —— 新建，不修改

到了第三天，才开始给 agent 很有限的“写”能力。这里我会严格限制：先允许 create，不允许 update/delete。

也就是说，优先开放：

  • 创建一个 task；
  • 创建一个 calendar event；
  • 上传一个文件；
  • 创建一份草稿文档。

先不开放：

  • 删除邮件；
  • 批量改日历；
  • 修改共享权限；
  • 自动发送消息；
  • 自动回复邮件。

原因很简单：创建类动作最容易审查，也最容易回滚。删改类动作一旦错了，恢复成本会高很多。

4) 第四天：先接通一条最小闭环 —— Gmail → Tasks

第一周里，我会最先正式启用的 workflow 是 Gmail → Tasks。原因不是它最厉害，而是它最像一个有真实价值、但又不太危险的闭环。

我会怎么设规则

  • 只看未读或我指定标签下的邮件；
  • 只提取“需要行动”的邮件；
  • 只生成 task，不自动回信；
  • task 标题尽量短；
  • task 备注里保留邮件来源线索。

我会怎么试运行

  • 先手动触发，不做定时；
  • 每天只跑 1 次；
  • 跑完人工 review；
  • 确认两三天稳定后，再考虑半自动。

这一步的目标不是“自动处理收件箱”，而是证明 agent 能把输入变成行动建议。

5) 第五天：加上 Calendar agenda + insert

Gmail → Tasks 稳一点之后，再加第二条：读 agenda + 新建 event。

这一天我只会允许它做两件事：

  • 每天生成 agenda 摘要；
  • 在我明确说“加一个事件”时，创建新的 calendar event。

我不会让它自动改已有会议，也不会让它自动发邀请。因为会议通常牵涉别人，一旦写错，社交成本会比技术成本更高。

最小成功标准

  • 能清楚列出今天日程；
  • 能按我说的时间建一个事件；
  • 时间、时区、标题没有明显偏差；
  • 遇到模糊描述时，先问而不是乱写。

6) 第六天：People + Drive，只做“最后一米”辅助

到了第六天，我会给它一点协作能力，但仍然保持轻量：

  • 查联系人；
  • 找文件；
  • 上传文件；
  • 生成共享链接。

这条线我会刻意限制在“最后一米”：

  • 不自动群发；
  • 不自动改共享权限；
  • 不自动通知所有人；
  • 先把找人、找文件、拿链接这几个动作做顺。

很多时候，系统不需要更聪明，只需要把最后一米少折腾一点。

7) 第七天：做一次回顾，只回答三个问题

第一周结束时，我不会急着再加技能。我只会问三个问题：

  1. 哪一个动作最稳定？
  2. 哪一个动作最常用？
  3. 哪一个动作一旦出错最麻烦？

这三个问题能决定第二周怎么扩。不是“还能接什么”，而是“什么值得留下”。

8) 如果是我，我会把权限分成三层

为了让第一周尽量稳，我会把 agent 能力分成三层：

Level 1：Read only

  • 读邮件
  • 读日程
  • 读任务
  • 查联系人
  • 列文件

Level 2：Create only

  • 建 task
  • 建 event
  • 上传文件
  • 创建草稿

Level 3：Sensitive write

  • 删改邮件
  • 改共享权限
  • 自动回信
  • 批量改日历
  • 自动通知他人

第一周只碰 Level 1 和 Level 2，不碰 Level 3。

9) 我会怎么写第一周的 agent 提示词

如果要给第一周的 agent 一个很短的 operating rule，我会写成这样：

先读取真实状态，再建议下一步；仅在用户明确要求或工作流已被批准时执行创建类动作；遇到模糊时间、模糊对象、模糊权限时必须先确认；不要自动删除、修改或发送。

不用很长，关键是边界清楚。

10) 一句话总结

第一周不是为了证明 agent 什么都能做，而是为了证明它能在很小的边界里，持续做对几件真的有用的事。

只要第一周能稳定做到：

  • 看邮件；
  • 转任务；
  • 看日程；
  • 建事件；
  • 查联系人；
  • 找文件；

那就已经不是 demo 了，而是一个开始接触真实工作的 agent。
