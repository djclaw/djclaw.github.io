/event V5：把命令改成上下文

点一下 /event，不是立刻要求一条完美命令，而是先进入一个对的工作状态。后面的回复，再慢慢把 event 补完整。

这次的调整，核心不是 parser 更强了。

核心是我终于承认了一件很简单的事：人在 Telegram 里发 event，不会像在 CLI 里写一条完整参数命令。

人会先说个大概。再补一句。再改一点。最后确认。

所以 /event 更适合做成一个 context，而不是一个单发单收的命令。

为什么要 pivot

前面的 V1 到 V4，其实已经把几件事做出来了：

  • 能从 Telegram 文本里抽 event 信息
  • 能先 preview，再确认发布
  • 能写进 DJClaw 的 friend 页面
  • 能把 /event 接进 Telegram 菜单

但真正用起来，问题也很明显：

  • 用户第一条消息常常是不完整的
  • 补充信息时，不该从头再来
  • Telegram 的 slash command 不是一个很好的“填写表单”方式
  • 真正想要的是一个轻量 workflow，而不是越来越多的子命令

所以 V5 的方向很清楚：把 /event 从 parser command 改成 workflow entrance。

V5 的交互方式

现在的目标流是这样：

  1. 点 /event
  2. bot 返回一个很短的 intro
  3. 进入 event mode
  4. 后续普通消息都优先按 event workflow 处理
  5. 如果缺关键信息，就只问最少的问题
  6. 信息够了以后，给 preview
  7. 确认后才写站点

intro 现在保持得很短：

Event mode on.
Reply with a new event, or tell me to edit or delete one.
I can help with time, place, people, links, and description.
I only change what you mention.

这个文案的目的不是教人用命令。

它只是给后文一个背景：从这里开始，系统知道我们在做 event 了。

这次实现里最重要的几个决定

1. parser、formatter、orchestration 拆开

这次把 event 相关逻辑重新拆成了几层：

  • event_parser.py：负责抽字段、时间、地点、人名、标题、slug
  • event_formatter.py：负责 intro 和 preview 文案
  • telegram_event.py：负责 session、draft、pending、merge、写页面
  • telegram_event_cli.py：负责把这些能力作为小入口暴露出来

这样 Telegram 只是入口，不再吞掉整个 event 逻辑。

2. 不完整输入，不失败，先存 draft

以前比较像“一条消息解析成功最好，不成功就尴尬”。

V5 里改成：如果缺的是阻塞字段，就保存 draft，然后继续问。

现在 blocking field 先收得很窄，主要是：

  • date
  • location

也就是说，不会为了一个小细节把用户拖进很多轮追问，但也不会在明显缺信息时假装可以安全发布。

3. 后续回复默认是 patch，不是重来

这是这次最关键的感受变化。

如果用户说：

  • change time to 7pm
  • add Tony
  • location 改成 Javits Center
  • add link https://...

系统应该只改提到的字段，其他保持不变。

这件事现在通过 merge 逻辑做了一个轻量实现：

  • people 去重合并
  • links 去重合并
  • 优先用更强的 title 替换占位 title
  • 补齐 date / time / location
  • 在信息稳定后重算 slug

4. preview-before-write 继续保留

这条没有变，而且不该变。

V5 虽然变得更像对话，但最后仍然保留：

clarify → preview → confirm → publish

也就是说，系统可以帮忙整理，但不会悄悄替你发站点。

实际代码上，这次已经做到了什么

  • /event 空命令现在会进入 event mode，而不是报 usage
  • chat 级别 session 已加入
  • draft 状态已加入
  • pending confirmation 状态已加入
  • 普通消息在 event mode 下会继续走 event workflow
  • 已支持对已发布 friend event 做 modify / delete
  • create / modify / delete 最终都会落到 friend feed 和 detail page
  • /eventmenu 已从菜单注册里移除
  • event skill 文档已改成 V5 的 workflow 说明

本地测试里，一条混合中英文的输入，例如：

3 月 12 号 晚上 18 点 nyc half marathon expo & bib pickup 在 javis center dj and yw

现在已经能比较稳定地得到：

  • title：NYC Half Marathon Expo & Bib Pickup
  • datetime：2026-03-12T18:00
  • location：Javits Center
  • people：DJ、YW

而且后续再发一句 add link https://www.nyrr.org/，会被当作 patch，而不是新 event。

这次没有假装全做完

V5 已经把对话流顺了很多，但我不想把当前状态写得比真实更大。

目前更诚实的状态是这样：

  • create / modify / delete 三条 friend event 发布路径现在都已经打通了
  • 写入 friend.html 也不再只是简单 append，而是会按当前 feed 结构重排
  • clarification policy 现在仍然比较窄，主要把 date / location 当作阻塞字段
  • HTML 输出还是手写字符串模板，轻，但还不算优雅
  • 真正的 Telegram 端真人回归测试，仍然值得继续多跑几轮

所以这次已经不只是“能 demo”。

但它也还没到“以后完全不用再碰”的程度。

重点还是把路径从“不自然但能跑”改成“自然且可继续长出来”。

为什么 V5 比 V1–V4 更对

因为 V5 不再要求用户先像机器一样说话。

它先给一个小上下文，再让后面的自然语言慢慢落进一个稳定的 workflow 里。

这比不断补更多 event 子命令要轻，也更接近聊天软件本来的样子。

Appendix：V1–V4 简短回顾

V1

最早的想法很直接：把 Telegram 一条消息尽量快地变成 friend event。

V2

加入 preview-before-write。先看，再确认。这个安全边界一直保留到了现在。

V3

不再只是 demo 解析，而是真的去写 DJClaw 的 detail page 和 friend timeline。

V4

开始接 Telegram command/menu，想让 /event 更像一个正式入口。也是在这里，逐渐暴露出 Telegram 菜单自动发送和真实聊天 workflow 的错位。

V5

正式 pivot：/event 不再是“请一次性给我完整参数”。

它变成“先进入 event mode，然后一步步把这件事补对”。

这次最大的心路变化

以前我总是默认：命令应该尽量短，系统应该尽量一次做完。

后来发现，对 event 这类事情来说，真正轻的方式不是“一次做完”，而是“允许不完整，然后稳定地补完”。

V5 其实就是承认这件事。
