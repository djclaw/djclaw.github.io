A quiet /event flow for Telegram

A small path. One message in Telegram. One clean event on the friend page. No dashboard. No heavy form. Just enough structure to keep it steady.

There was a simple need behind this.

Sometimes an event starts as one rough message on the phone: a date, a place, two names, maybe a link. That is usually the real shape of the note. Fast. Incomplete. Human.

So instead of building a heavy admin page, I built a lighter path. Telegram stays as the input surface. Nanobot turns that message into a preview. Then one tap decides it: write it to DJClaw, or drop it.

What this flow does

The flow takes one raw Telegram message and turns it into a friend event entry.

  • Parse the raw event text
  • Extract date, time, title, people, place, and links
  • Generate a preview
  • Save a temporary pending record
  • Wait for confirmation
  • Write the detail page and update friend.html

The important part is the pause in the middle. Nothing gets published immediately. First comes a preview. Then a yes or no.

The files behind it

The core implementation lives in two files:

  • /home/dj/.nanobot/workspace/scripts/telegram_event.py
  • /home/dj/.nanobot/workspace/scripts/telegram_event_cli.py

The first file does the real work. The second one gives that work a small command line wrapper so the bot loop can call it cleanly.

telegram_event.py

This is the engine.

It handles five quiet jobs:

  1. Parse input. It looks through raw text and tries to find dates, times, links, names, locations, and optional explicit titles.
  2. Normalize output. It turns loose text into a structured event object with fields like title, datetime_iso, location, people, description, and slug.
  3. Create a safe preview. It builds a readable confirmation block so the event can be checked before anything is written.
  4. Render HTML. It creates the event detail page and the event card that appears in the friend feed.
  5. Update the site. It writes the new detail page and patches friend.html so the event appears in the correct order on the timeline.

telegram_event_cli.py

This is the small front door.

It exposes three commands:

  • preview
  • confirm
  • cancel

That separation matters. The bot or channel layer does not need to know how HTML is generated. It only needs to call a simple command and read the result.

How it is used

The basic rhythm looks like this:

  1. Send raw event text from Telegram
  2. Run preview to parse it
  3. Read the preview
  4. Tap confirm or cancel
  5. If confirmed, write the event into DJClaw

A rough example input could be as simple as:

Mar 22 2026 1 PM with DJ, ZZ, Tony at Yankee Stadium https://example.com title: Messi at Yankee Stadium

From that, the parser can infer:

  • date
  • time
  • people
  • location
  • link
  • title
  • a final output filename slug

How the preview step works

The preview command does not write to the site yet.

Instead, it creates a token from chat id, message id, and raw text. That token becomes a tiny temporary handle for the pending event. The event payload is saved as JSON under:

/home/dj/.nanobot/workspace/tmp/telegram-event-pending/

This is the quiet safety layer. If the draft looks wrong, cancel it. If it looks good, confirm it. Nothing is forced too early.

How confirmation works

Once confirmed, the CLI loads the saved pending payload by token and calls the write path.

That write path does two things:

  1. Create the new friend event detail page inside the DJClaw site
  2. Update friend.html so the event shows up in the main feed

After the write succeeds, the pending JSON is deleted. The system goes back to clean.

What gets created or modified

When one event is confirmed, the usual file changes are:

  • Create one new event page in /home/dj/.nanobot/workspace/djclaw.github.io/
  • Modify /home/dj/.nanobot/workspace/djclaw.github.io/friend.html
  • Create and later remove one temporary JSON file in tmp/telegram-event-pending/

The generated event page includes:

  • a calendar badge
  • status
  • title
  • summary
  • time
  • place
  • people
  • links

The friend feed card includes:

  • a timeline rail
  • a date badge
  • upcoming or past status
  • title
  • a compressed summary line

How ordering is handled

The feed is not just append-only.

Existing event cards are parsed from friend.html, the incoming event is inserted, and then the whole list is sorted again.

The logic is simple:

  • upcoming events first
  • upcoming events in forward time order
  • past events after that
  • past events in reverse time order

So the page keeps its shape without needing a database or admin panel.

Why this structure feels right

I like this because each layer stays small.

  • Telegram is just the entry point
  • the CLI is just a wrapper
  • the parser is just a parser
  • the site writer only writes site files

No part has to carry the whole system.

That means the flow is easier to patch, easier to read later, and easier to move if the channel changes one day.

The internal call shape

In plain language, the call chain is this:

Telegram message
  ↓
channel or bot action
  ↓
telegram_event_cli.py preview
  ↓
telegram_event.py parse + save pending + build preview
  ↓
user taps confirm
  ↓
telegram_event_cli.py confirm
  ↓
telegram_event.py write_event()
  ↓
friend.html updated + detail page created

That is the whole loop.

Short enough to understand. Structured enough to trust.

A small note on style

This system is not trying to be smart in a loud way.

It is just trying to reduce friction.

One message in. One preview. One calm choice. Then the page updates.

That is enough.
