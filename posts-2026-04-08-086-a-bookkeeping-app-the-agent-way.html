A Bookkeeping App, the Agent Way

Not a full accounting system. Just a smaller tool where the agent does the tedious first pass and the human reviews exceptions.

Once the pipeline became clear, I wanted to test it on something practical. A bookkeeping app felt like a good candidate because the pain is obvious and the work is repetitive.

Most small bookkeeping tools still ask the user to become a part-time clerk. Upload things. Clean things. Rename things. Categorize things. Double-check everything. Repeat next month.

I wanted to try a different center of gravity.

The user

The product is for one-person operators:

  • freelancers,
  • consultants,
  • creators,
  • small service businesses,
  • side-hustle owners.

These people usually do not want an enterprise accounting suite. They want their records to stop feeling messy. They want taxes to feel less scary. They want to know where the money went.

The product promise

The MVP promise is intentionally small:

Upload a transaction file, let the system categorize what it can, review the uncertain items quickly, and export a tax-ready summary.

That is enough to be useful. It does not need payroll, invoices, reconciliation, or a giant dashboard on day one.

The key shift

The main idea is simple: the app should not behave like a static ledger with a chatbot glued onto it. The workflow itself should assume that the first pass is done by the agent.

So the rhythm becomes:

  1. import transactions,
  2. normalize names and fields,
  3. suggest categories,
  4. flag uncertainty,
  5. let the human review exceptions,
  6. export something clean.

In other words: the user should not do all the boring sorting by hand. The user should mainly supervise the edge cases.

What belongs in the MVP

  • manual CSV import,
  • transaction normalization,
  • suggested categorization,
  • a small category system,
  • a review queue for low-confidence items,
  • monthly summary,
  • CSV export,
  • a simple command-style interface for bookkeeping tasks.

And just as important, what does not belong:

  • bank connections,
  • payroll,
  • full collaboration,
  • advanced journal entry workflows,
  • a big accounting-language surface.

That boundary is what keeps the idea honest.

The core screens

If I were sketching the first version, I would keep the interface very lean.

1. Home / Inbox

This is not a dashboard for showing off charts. It is a place to answer one question: what needs attention right now?

2. Agent Console

This is the center. The user can say things like:

  • Import this March CSV and categorize what you can.
  • Show me only uncertain items.
  • Reclassify all Figma charges as Software.
  • Give me a Q1 tax-style summary.

The point is not to make it magical. The point is to make the workflow legible.

3. Transactions List

A clean table with date, merchant, amount, category, confidence, status, and notes. This is where the bookkeeping becomes visible again.

4. Review Queue

This may matter more than a dashboard. Ambiguous categories, likely duplicates, likely transfers, and odd transactions should all gather here. The user should be able to clear the queue quickly.

5. Categories / Rules

A small place to manage the logic that makes future months easier.

6. Reports / Exports

Nothing fancy. Just enough to answer practical questions and produce a clean handoff.

The design principle underneath it

The deeper principle is not "AI does bookkeeping for you." That is too vague, and too dangerous.

The principle is this:

let automation do the first pass, but make every financial decision reviewable, reversible, and explainable.

That means the system can suggest, group, flag, and prepare. But it should not silently harden doubtful bookkeeping decisions into the record.

Why I like this as a product example

I like this example because it shows where agent-first software can actually feel honest.

The goal is not to pretend that accounting disappears. The goal is to move the human to the right part of the loop.

Instead of spending an hour sorting clean transactions and five minutes on weird ones, the user should spend five minutes reviewing the weird ones and let the system carry the repetitive load.

The one-line version

If I had to compress the product into one sentence, it would be this:

A lightweight bookkeeping app where the system does the first-pass cleanup of transactions, and the user only reviews exceptions before exporting tax-ready records.

That feels small enough to build, and real enough to matter.
