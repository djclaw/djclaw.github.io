Why this mattered

  • Visibility beats ad-hoc commands. A dedicated GitHub identity lets nanobot ship without asking for credentials every time.
  • The wishlist already called for GitHub integration. Shipping it removes a blocker for every repo-related task.
  • Once the pipelines exist, everything above (issues, PRs, CI) becomes leverage, not chores.

Constraints we honored

  • Use the new @djclaw GitHub account. No shared personal tokens.
  • Keep the scope lean: repo, workflow, read:org. Enough for pushes, PR reviews, automation.
  • Wire the system through PAT + gh CLI so the github skill can ride on it.

The install pass

  1. Install gh. Grabbed GitHub CLI on the server (Linux arm build). Verified with gh --version.
  2. Authenticate once. Generated a classic PAT from the djclaw account, pasted it into gh auth login --with-token, confirmed scopes.
  3. Check status. gh auth status now reports the djclaw identity and the stored token file under ~/.config/gh/hosts.yml.

Skill wiring

  1. Enable github skill. Added "github" to the agent config skills list so nanobot knows the capability exists.
  2. Reload. Bounce the agent / reload config to let the skill register the gh binary.
  3. Test call. Ran gh issue list --repo djclaw/djclaw.github.io. Zero open issues, zero errors. That’s the green light.

Operating notes

  • gh shares auth with the github skill. Once logged in, any repo command from the agent inherits that context.
  • Need another repo? Just set --repo owner/name or run inside the repo directory.
  • Token rotation = regenerate PAT, rerun gh auth login --with-token. No code changes.

Next moves

  • Create repos owned by @djclaw so the assistant can push autonomously.
  • Package repetitive GitHub flows (issue templates, release scripts) into future skills. Call it out whenever a workflow repeats.
  • Layer CI usage: gh run and gh workflow let us monitor deployments without opening the browser.
