gws 四层结构图：tool、skill、workflow、recipe

如果只想先把脑子理顺，先看树状图。然后再看下面每一层到底在解决什么问题。

刚才我自己也在重新整理这件事：如果拿 Google Workspace CLI，也就是 gws，来举例，tool、skill、workflow、recipe 到底应该怎么分？

我现在觉得，最顺的一种看法是：先分四层，再把 recipe 看成最上面那层里的另一种形状。

先看最短树状图

gws
├─ tool / 原始能力
│  ├─ 读 Google Sheet
│  ├─ 创建 Calendar event
│  ├─ 发 Gmail
│  └─ 建 Google Doc
│
├─ foundational skill / 基础 skill
│  ├─ gws-sheets-read
│  └─ gws-calendar-insert
│
├─ domain skill / 领域伞形 skill
│  ├─ gws-sheets
│  ├─ gws-calendar
│  └─ gws-drive
│
└─ workflow layer / 任务层
   ├─ workflow skill
   │  └─ gws-workflow-weekly-digest
   └─ recipe skill
      └─ recipe-create-events-from-sheet

如果只说一句：

tool 是零件，基础 skill 是顺手的小工具，领域 skill 是地图，workflow / recipe 是做完整任务的方法。

1. tool / 原始能力

这是最底层，是真正干活的接口。

比如：

  • 读 Google Sheet
  • 创建 Calendar event
  • 发 Gmail
  • 建 Google Doc

这些更像是零件。

单看每一个都很有用，但如果直接把一大堆原始能力摊在 agent 面前，agent 往往会遇到三个问题：

  • 不知道先选哪个
  • 知道方向，但不确定参数怎么拼
  • 会做单步动作，但不会把几步串成任务

所以 tool 很重要，但它通常不是最终给 agent 用的最好形态。

2. foundational skill / 基础 skill

这一层就是：把一个常见动作包一下，让它更顺手。

比如：

  • gws-sheets-read：专门读表格
  • gws-calendar-insert：专门创建日历事件

这类 skill 的特点通常很稳定：

  • 一件事
  • 名字很清楚
  • agent 不容易选错

所以如果你只是想说：

把这个表读出来。

那直接用 gws-sheets-read 就够了。

它的价值不是“功能最多”，而是“少想一步”。

3. domain / umbrella skill

这一层是一个领域的总入口。

比如：

  • gws-sheets
  • gws-calendar
  • gws-drive

它不是只做一个动作，而是告诉 agent：

  • 这个领域能干什么
  • 有哪些子命令
  • 怎么查 schema
  • 什么时候该用简单 helper，什么时候该用更底层方法

比如你说：

我不只是要读 sheet，我还可能要 append、update、format。

那这时候就更像该走 gws-sheets 这种伞形 skill。

所以这一层更像地图，不是按钮。

4. workflow skill

workflow skill 是“已经打包好的整套流程”。

比如一个假想的 gws workflow：

  • gws-workflow-weekly-digest

它内部可能已经默认好了：

  1. 读本周日历
  2. 汇总任务
  3. 生成文档
  4. 发给指定对象

这时候你不是在自己拼动作，而是在调用一个比较完整的成品流程。

所以 workflow skill 更像：

一键做完一件常见工作。

它适合的场景通常是：

  • 流程很稳定
  • 输入输出很固定
  • 经常重复
  • 适合一键执行

例如：

  • 每周 agenda 汇总
  • 固定格式报告生成
  • 固定模板邮件发送

5. recipe skill

recipe skill 和 workflow 很像，但关键区别是：recipe 更像方法，不一定是一个现成的成品按钮。

比如：

  • recipe-create-events-from-sheet

它的意思更像：

  1. 先从 sheet 里读取活动数据
  2. 检查标题、时间、地点字段
  3. 逐条创建 calendar events
  4. 写入前确认
  5. 失败时记录错误项

这里重点不是“有一个神奇按钮”，而是：

这是一套 agent 应该怎么做的步骤。

所以 recipe 更像“做法”，workflow 更像“成品”。

用一个最具体的对比

场景 A：你说——

把这个 Google Sheet 里的活动导入日历。

如果是 workflow skill

可能会长这样：

  • gws-workflow-import-sheet-events

它的特点是：

  • 输入 sheet 链接
  • 自动处理字段映射
  • 自动创建 event
  • 基本可以直接跑

这像一个现成功能。

如果是 recipe skill

可能会长这样：

  • recipe-create-events-from-sheet

它的特点是：

  • 先读表
  • 再检查列结构
  • 再决定时间格式
  • 再逐条写入 calendar
  • 写入前提醒确认

这更像标准操作流程。

再说得更白一点

workflow skill 像什么？
像咖啡机上的一个按钮：

  • 美式
  • 拿铁
  • 卡布奇诺

按下去，它自己完成很多步骤。

recipe skill 像什么？
像一张咖啡配方卡：

  • 先磨豆
  • 再萃取
  • 再打奶泡
  • 最后按比例混合

它不是按钮，但它能指导你稳定做出来。

放回 gws 的语境里

你可以这样理解：

  • gws-sheets-read = 一个小动作
  • gws-sheets = 一个领域入口
  • gws-workflow-* = 已封装好的流程块
  • recipe-* = agent 可复用的执行方案

什么时候该做 workflow，不该只做 recipe？

适合做 workflow：

  • 流程很稳定
  • 输入输出很固定
  • 经常重复
  • 适合一键执行

比如：

  • 每周 agenda 汇总
  • 固定格式报告生成
  • 固定模板邮件发送

什么时候更适合 recipe？

适合做 recipe：

  • 中间常常要判断
  • 数据格式不总一样
  • 需要 agent 临场调整
  • 有时要人工确认

比如：

  • 从杂乱的表里提取活动并导入日历
  • 根据邮件内容判断是否转任务
  • 从多份文档整理一份 summary

一句话版

在 gws 里：

  • workflow skill = 已经封装好的可执行流程
  • recipe skill = 教 agent 怎么把多个 gws 能力串起来完成任务

如果再往前补一句，那就是：

tool 解决“能不能做”，skill 解决“怎么更顺地做”，workflow 和 recipe 解决“怎么把事情真正做完”。
