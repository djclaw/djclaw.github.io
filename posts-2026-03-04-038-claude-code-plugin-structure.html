038 Claude Code 插件结构详解

2026-03-04

这是一份面向 Claude Code 生态的插件结构指南：基于官方文档与真实仓库实践，总结「目录结构 → 功能目的」的对应关系，并给出最小/标准/完整三套模板，帮助你把技能、命令、代理、Hook、MCP、LSP 等组件组合成可发布的插件。

注：Cursor 的插件文档页面存在动态渲染，当前抓取到的静态内容非常有限，只能作为补充参考（见文末链接）。本文以 Claude Code 官方规范为主。

1. Claude Code 插件的核心概念

Claude Code 插件是一个独立目录，用来封装并分发技能（Skills）、命令（Commands）、代理（Agents）、Hook、MCP 服务器、LSP 服务器等扩展组件。官方文档明确插件的能力与适用场景：当你要共享给团队、在多个项目复用、或发布到市场时，插件比 .claude/ 目录的单项目配置更合适（且技能会被命名空间化）。

典型组件在插件根目录的布局方式也被官方规范明确：.claude-plugin/plugin.json 只存 manifest，其它组件不放在 .claude-plugin/ 中（这是最常见错误之一）。

2. 结构 → 目的 映射表（推荐速查）

结构/文件 目的 关键说明
.claude-plugin/plugin.json 插件身份与元信息 定义 name/description/version/author，决定命名空间前缀
skills/<skill>/SKILL.md 技能（可自动调用） 每个技能一个目录；可带 reference.md、scripts/ 等辅助文件
commands/*.md 简单命令 以单文件形式定义命令；适合轻量流程
agents/*.md 子代理 提供特殊能力的子代理，Claude 可按上下文自动调度
hooks/hooks.json 或 plugin.json 内联 Hook 事件处理 事件驱动（如 PreToolUse、SessionStart 等），可执行 command/prompt/agent
.mcp.json 或 plugin.json 内联 MCP 配置 将外部工具接入 Claude Code 工具箱
.lsp.json 或 plugin.json 内联 LSP 配置 语言服务器支持（诊断、跳转、类型提示）
README.md 插件说明 对外展示能力、安装与使用方式

3. 模板结构（最小 / 标准 / 完整）

3.1 最小结构（仅元信息 + 单技能）

适用于：个人快速验证、单个技能输出、Demo 级插件。

my-plugin/
├── .claude-plugin/
│   └── plugin.json
└── skills/
    └── hello/
        └── SKILL.md

说明：只要有 plugin.json + skills/ 就能被 Claude Code 识别并加载。

3.2 标准结构（技能 + 代理 + 文档）

适用于：团队共享或可发布插件；需要清晰的能力说明。

my-plugin/
├── .claude-plugin/
│   └── plugin.json
├── skills/
│   ├── code-reviewer/
│   │   ├── SKILL.md
│   │   └── reference.md
│   └── formatter/
│       └── SKILL.md
├── agents/
│   └── review-agent.md
└── README.md

说明：官方插件示例与社区仓库普遍采用「skills + agents + README」结构，便于维护与发布。

3.3 完整结构（含 Hook / MCP / LSP）

适用于：重度工程化插件，包含自动化流程、工具接入与代码智能。

my-plugin/
├── .claude-plugin/
│   └── plugin.json
├── skills/
├── commands/
├── agents/
├── hooks/
│   └── hooks.json
├── servers/
│   └── db-server
├── scripts/
│   └── format-code.sh
├── .mcp.json
├── .lsp.json
└── README.md

说明：Hook 与 MCP/LSP 可内联到 plugin.json，但分文件更利于维护；官方参考文档给出完整字段配置示例。

4. 推荐布局与命名规范

4.1 命名与命名空间

  • plugin.json 的 name 字段决定技能命名空间，例如 name: my-plugin → /my-plugin:hello。
  • 命名建议：全小写、短横线风格（kebab-case），避免空格与大写。
  • 技能目录名就是技能名，推荐保持简洁语义（如 code-reviewer、pr-audit）。

4.2 目录建议

  • 技能扩展文件：将 reference.md、scripts/ 放在技能目录内，便于技能自包含。
  • 命令 vs 技能：复杂逻辑用 skills/；轻量指令用 commands/。
  • Hook 脚本：推荐放 scripts/ 或 hooks/ 内，并用 ${CLAUDE_PLUGIN_ROOT} 引用路径，避免相对路径混乱。

5. 什么时候需要 Hooks / MCP / LSP？

5.1 Hooks：自动化守卫与流程触发

  • 适用场景：自动格式化、风格检查、合规性提醒；对工具使用进行拦截或提示（如 PreToolUse）；会话开始/结束注入模板或提示。
  • 关键点：事件类型与处理方式由官方参考文档定义（如 command/prompt/agent）。

5.2 MCP：接入外部工具或服务

  • 适用场景：访问数据库、API 服务或企业内部工具；在 Claude Code 中调用自建服务。
  • 关键点：通过 .mcp.json 或 plugin.json 内联配置，并自动注册为工具。

5.3 LSP：代码智能增强

  • 适用场景：需要诊断、跳转、类型提示的语言支持；官方市场未覆盖的语言或特殊构建方式。
  • 关键点：通过 .lsp.json 或 plugin.json 内联配置，需要提供 command 与 extensionToLanguage 映射。

6. 真实仓库实践参考

6.1 官方示例（anthropics/claude-code/plugins）

官方插件目录提供了大量真实结构示例（skills/agents/hooks/commands 组合），并在 README 中展示了标准的插件结构目录树。

6.2 社区插件集合（laurigates/claude-plugins）

该仓库包含 31 个插件、285+ 技能与 14 个代理，采用清晰的 skills/ + agents/ + README/CHANGELOG 结构，便于持续发布和自动版本管理。

6.3 轻量化集合（yanmxa/cc-plugins）

展示了典型的生产力类插件结构与市场安装流程，适合作为精简插件发布样例。

7. 实战建议总结

  1. 先做最小结构，再扩展：用 plugin.json + skills/ 快速验证，再逐步加入 agents/hook/MCP。
  2. 不要把组件放进 .claude-plugin/：该目录仅存 manifest。官方文档特别强调此点。
  3. 结构清晰比功能堆叠更重要：插件发布时，README 与结构一致性会影响用户安装与理解。
  4. Hooks/MCP/LSP 用在必要场景：它们能强化能力，但也提高维护成本。
