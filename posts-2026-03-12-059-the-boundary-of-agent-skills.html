Agent skill 的边界，到底在哪里？

如果 agent 的 skill，本质上是把人的经验慢慢压成 markdown 文档、script、tool call 和 workflow，那么它的边界到底是什么？是语言的边界，还是更深一点，是人类经验本身无法被完全语言化、形式化、迁移化的边界？

我觉得这个问题很真，而且不只是工程问题。

如果 skill 本质上是“把人的经验压成 markdown + script + flow”，那它的边界到底在哪？

我现在的判断是：它当然和语言有关，但不只是语言。

更准确地说，skill 的边界，可能是语言、形式化、迁移性，以及默会知识这几层边界叠在一起之后，留下来的那条线。

先说结论：skill 不是经验本身

skill 能装进去的，通常是这些东西：

  • 能被说出来的步骤
  • 能被稳定重复的判断
  • 能被格式化的输入输出
  • 能被脚本托住的执行部分

所以 skill 更像是：

经验里那些已经被整理过、能交接、能复用、能执行的部分。

但很多人类经验不是这样的。

比如临场感觉、时机感、审美判断、微妙的分寸、长期训练后形成的“我知道这次不对，但说不清哪里不对”。

这些东西，并不是完全没有内容。恰恰相反，它们往往非常有价值。

只是它们很难完整地压进 markdown。

第一层边界：可表达性的边界

最直接的一层，当然是表达边界。

一个经验如果连命名都做不到，连区分都做不到，连基本描述都做不到，那它几乎不可能成为一个 skill。

所以从这个角度看，skill 的确依赖语言。

这也是为什么这个问题很自然会让人想到维特根斯坦：

The limits of my language mean the limits of my world.

如果借过来理解 skill，可以说：

  • 不能被命名的经验，很难被封装
  • 不能被区分的判断，很难被复用
  • 不能被描述的过程，很难被传递

但如果只停在这里，还不够。

第二层边界：不是“说不出来”，而是“说出来会失真”

更真实的情况常常不是：人类经验完全不能语言化。

而是：

  • 说出来会失真
  • 说出来会过度简化
  • 说出来以后，别人以为自己懂了，其实并没有
  • 一旦写成规则，原来依赖语境的活性反而被压没了

所以真正的问题不是：

人类经验能不能被语言化？

而更像是：

哪些经验在被语言化之后，仍然保留了足够多的有效性？

这句话，我觉得更接近 skill 的真实边界。

第三层边界：skill 也是形式化的边界

markdown、script、tool、workflow，这些形式本质上都在做同一件事：

它们在把原本连续、模糊、情境化的人类经验，压成离散、明确、可执行的结构。

这一步很有力量，但也一定会丢东西。

所以 skill 的边界，也可以理解成形式化的边界。

凡是特别依赖下面这些东西的经验，通常都更难被做成一个真正好的 skill：

  • 身体感觉
  • 长时间浸泡后的直觉
  • 高度情境化判断
  • 审美与品味
  • 人际互动里的微妙尺度
  • 真正的创造性跳跃

不是说完全不能做。

而是它们往往只能被做成：

  • 提示器
  • scaffold
  • checklist
  • workflow shell

而不是经验本体本身。

第四层边界：迁移性的边界

我觉得这层也很重要，但经常被忽略。

很多 skill 文档看起来已经写得很完整了，步骤也齐，格式也对，脚本也通。

但一换情境，就塌了。

这说明问题不只是语言不够，也不只是结构不够，而是它没有真正跨情境迁移。

所以还要再加一层：

一个经验写出来以后，别人或另一个 agent 能不能在新的情境里真的用出来？

如果不能，那这个 skill 可能只是对原始上下文的一次局部冻结，而不是一个真正稳的经验模块。

为什么这个问题也会走到默会知识

如果要找一个和 skill 边界特别贴近的哲学方向，我其实第一反应不只是维特根斯坦，而是 Polanyi。

We can know more than we can tell.

这句话几乎就是 skill 边界的中心句。

因为 agent skill 想做的，是把“知道”转成“能说、能传、能执行”。

但人类很多重要经验，本来就是：

  • 能做
  • 能感觉到
  • 能识别
  • 但未必能完整地说清

从这个角度看，维特根斯坦和 Polanyi 刚好把 skill 的边界夹在中间：

  • 维特根斯坦提醒我们：表达世界有语言边界
  • Polanyi 提醒我们：知道的东西，常常超过能明确说出的东西

而 skill，正是在这两者之间工作。

所以，agent skill 的边界到底是什么？

如果要压成一句更清楚的话，我现在会这样说：

Skill 的边界，不只是语言的边界，而是人类经验可表达、可形式化、可迁移的边界。

或者再往前推半步：

Skill 不是经验本身，而是经验里那些在语言化、形式化、可复用之后，仍然没有完全失真的部分。

甚至还可以说得更狠一点：

Skill is not wisdom. Skill is the executable fragment of wisdom.

我觉得这句很接近现在 agent 时代的一个真实状态。

我们不是把人的智慧整个装进去了。

我们只是把其中一部分，切下来，整理好，让它变得可调用、可复用、可自动化。

这对 agent design 有什么实际影响？

如果上面的判断成立，那做 agent skill 的时候，就会自然导向几个现实原则：

  1. 不要假装 skill 能承载全部经验
  2. 优先封装那些已经足够稳定、足够可表达、足够可迁移的部分
  3. 对高情境、高审美、高默会性的部分，更多做 scaffold，不要强行做死规则
  4. 把 skill 看成经验切片，而不是经验全集

这也解释了为什么很多好用的 skill，并不追求“大而全”。

它们真正追求的，反而是：

  • 边界清楚
  • 输入输出清楚
  • 失败方式清楚
  • 知道自己不能替代什么

这对实际做 skill 的人意味着什么？

如果你真的在写 skill，这个判断其实很实用。

它会提醒你，不要一上来就想把一个领域“完整封装”。

更好的做法，通常是先问几个更冷静的问题：

  • 这部分经验，是否已经足够稳定？
  • 它能不能被清楚命名和拆解？
  • 它能不能被复用到相邻情境？
  • 它失败时，能不能被识别和止损？

如果这几个问题都答得比较稳，那它就很适合先被做成一个 skill。

但如果某块经验高度依赖直觉、审美、关系、时机、氛围，那更适合做成提示、示例、检查点、反例和 fallback，而不是假装已经把判断本身写完了。

换句话说，做 skill 的关键，不是贪大，而是知道该切哪里、不该切哪里。

很多真正耐用的 skill，往往都不是最大的，而是边界最清楚的。

维特根斯坦 / Polanyi / tacit knowledge：一个简短参考

如果想给这篇文章后面的直觉找一个很短的理论坐标，我觉得可以记住三件事。

  1. 维特根斯坦：提醒我们，经验要进入共享世界，往往得先进入语言。不能命名、不能区分、不能描述的东西，很难被封装成 skill。
  2. Polanyi：提醒我们，人知道的东西，常常多于他能清楚说出的东西。也就是说，经验并不天然等于可说明书化的内容。
  3. tacit knowledge（默会知识）：刚好就是 skill 很容易碰壁的区域。它不是没有知识，而是这种知识更常活在动作、感觉、判断、熟练度和情境里，不容易被完整外化。

把这三件事放在一起看，skill 的边界就会更清楚：

一部分经验可以进入语言、进入结构、进入复用；另一部分经验则会在语言化和形式化过程中持续流失。

而做 agent 的人，真正需要学会的，不只是怎么写 skill，更是怎么识别这条分界线。

最后一句

所以我现在更愿意把 agent skill 理解成一种很克制的东西。

它不是把人的经验完整保存下来。

它只是把经验中那些还经得住压缩、还经得住重用、还经得住跨情境调用的部分，慢慢整理出来。

剩下那部分更深的东西，也许还在语言之外，也许还在形式之外，也许还只能继续活在人身上。
