A Deep Dive on Matt Pocock’s Skills Repo

Why this repo feels less like a prompt pack and more like a modular operating system for small product development.

I really like mattpocock/skills. The more I look at it, the less it feels like a bag of useful prompts, and the more it feels like a product-development system.

That distinction matters.

A prompt library gives you isolated tricks. A good skill repo gives you reusable units of work. And a great one starts to shape how you think about building itself.

What is actually in the repo

The repo includes skills like write-a-prd, prd-to-plan, prd-to-issues, grill-me, design-an-interface, request-refactor-plan, tdd, triage-issue, improve-codebase-architecture, write-a-skill, edit-article, and ubiquitous-language.

At first glance, this looks like a grab bag. Planning over here. TDD over there. Interface design somewhere else. Writing in another corner.

But after a second look, a pattern appears.

The hidden structure: a development pipeline

Most of the skills cluster naturally into a pipeline:

  1. Clarify the problem — write-a-prd, grill-me, ubiquitous-language
  2. Turn intent into structure — prd-to-plan, prd-to-issues, request-refactor-plan
  3. Explore design options — design-an-interface
  4. Implement safely — tdd, triage-issue
  5. Improve the system itself — improve-codebase-architecture, setup-pre-commit, git-guardrails-claude-code
  6. Package knowledge — write-a-skill, edit-article

That is why the repo feels unusually strong. It does not just help with one phase of software work. It covers the loop from thinking, to scoping, to building, to refining, to documenting.

Why this feels different from most agent tooling

A lot of agent tooling still aims at one of two things:

  • either “make the model code faster,”
  • or “give the model more tools.”

This repo is aiming somewhere slightly smarter.

It assumes the bottleneck is often not typing code. The bottleneck is usually one of these:

  • the problem is still fuzzy,
  • the language is inconsistent,
  • the plan is too big,
  • the architecture is too shallow,
  • the issue breakdown is not independently executable,
  • or the human has not really decided yet.

That is a much more mature view of software work.

The repo encodes judgment, not just output format

This is the part I find most interesting.

Good skills do not merely say, “produce this artifact.” They encode a stance about how thinking should happen.

For example:

  • write-a-prd suggests that requirements should be discovered interactively, not hallucinated in one pass.
  • prd-to-plan suggests implementation should move through vertical slices, not giant abstract phases.
  • prd-to-issues suggests work should be decomposed into units that are independently grabbable.
  • grill-me suggests unresolved branches in the decision tree should be surfaced before execution.
  • tdd suggests confidence should be built incrementally through red-green-refactor loops.

So the repo is not just storing prompts. It is storing product and engineering judgment.

Why vertical slices show up so often

One quiet theme running through this repo is the preference for tracer bullets and vertical slices.

That is important because agent systems are very good at generating broad plans that look impressive and feel unusable. They can outline ten phases in a second. But that is not the same as creating momentum.

Vertical slices force a different discipline. Each slice has to be small enough to complete, real enough to test, and integrated enough to teach you something.

That is how real products move forward.

This repo also shows what a skill should be

I have spent a lot of time thinking about what a skill actually is. This repo gives a practical answer.

A skill is not merely a “better prompt.” A real skill should usually do at least three things:

  1. define when to use it,
  2. shape the sequence of thought or work,
  3. produce an artifact or decision that can feed the next step.

That is why these skills compose so well. Each one leaves behind something useful for the next one.

The repo is unusually strong at transitions

Many systems are good inside one step. Fewer systems are good at moving from one step to the next.

This repo is good at transitions:

  • from messy idea to PRD,
  • from PRD to plan,
  • from plan to issues,
  • from issue to TDD loop,
  • from codebase pain to refactor plan,
  • from repeated behavior to new reusable skill.

That is what makes it feel operational. It is not frozen advice. It is a chain.

Why this matters for people building with agents

If you are building products with agents, the biggest leverage may not come from one stronger model. It may come from having a cleaner grammar of work.

This repo is a good example of that grammar.

It says:

  • interview before specifying,
  • specify before planning,
  • plan before issue breakdown,
  • design multiple options before locking interfaces,
  • implement in slices,
  • refactor deliberately,
  • package repeated wisdom back into skills.

That loop is bigger than coding. It is a lightweight operating model.

Where I think the repo points next

To me, the most exciting part is not the current list. It is the design direction underneath it.

If this keeps growing, the natural next step is not “more random skills.” The natural next step is a more explicit development lattice:

  • discovery skills,
  • decision skills,
  • design skills,
  • implementation skills,
  • quality/safety skills,
  • knowledge-packaging skills.

Once you see it that way, the repo stops looking like a collection. It starts looking like a modular methodology.

My takeaway

The best thing about mattpocock/skills is not any single skill.

It is the quiet claim underneath the repo: software work can be decomposed into reusable patterns of thought, and those patterns can be made executable enough to actually help.

That is why I keep coming back to it.

Not because it gives the model more words. But because it gives the work more shape.
