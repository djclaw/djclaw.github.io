有了 MCP，还要不要再做一个 skill？

短答案：不一定。MCP 是能力，skill 是套路。

这两天重新想了一遍 MCP 和 skill 的关系，结论其实比想象里简单。

MCP 解决的是“我能做什么”。它把外部能力接进来，比如读日历、发邮件、查数据库、控制设备、调用 API。

Skill 解决的是“什么时候做、怎么做、按什么顺序做”。它更像一份小型工作流：触发条件、步骤、边界、输出格式、失败时怎么回退。

什么时候只用 MCP 就够了

如果任务很简单，通常不用再额外做一个 skill。

  • 问天气。
  • 查一个文件。
  • 读某条日历事件。
  • 执行一个非常直接的单步操作。

这类场景里，模型经常可以直接根据当前意图去调用 MCP。它不是靠机械关键词匹配，而是靠对任务的理解。

什么时候值得再做一层 skill

  • 任务有多个步骤。
  • 你想要固定输出格式。
  • 你总在重复解释同一套流程。
  • 这件事有前置检查、边界条件、回退逻辑。
  • 同一个 MCP 里有很多工具，模型容易用得不稳定。

比如一个 Calendar MCP 本身可以查日程、建活动、改活动。但如果你真正想要的是“生成 agenda digest”“按默认规则建 event 并邀请协作者”，那就很适合再包一层 skill。

只靠关键词触发，稳不稳？

不太稳。

现实里更接近这样：模型看到当前问题，理解用户意图，再决定要不要调用某个工具。关键词会有帮助，但它不是可靠的硬规则。用户换一种说法，工具名写得不清，或者上下文略复杂，结果就可能变。

所以如果你在意稳定性，不要把设计建立在“出现某个词就一定触发”上面。更好的做法是：

  • 把 MCP 的工具描述写清楚。
  • 把常用工作流做成 skill。
  • 让 skill 明确写出适用场景、步骤和边界。

我现在的判断方法

如果它只是一个明确能力，先上 MCP。

如果我已经开始反复告诉 agent 同一套做法，那就该补一个 skill 了。

简单说：

  • MCP：给 agent 手。
  • Skill：给 agent 路线。

有手，不代表一定会走得稳。要走得稳，往往还是需要路线。
