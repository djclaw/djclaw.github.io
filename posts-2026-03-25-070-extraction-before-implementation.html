Extraction：先看清，再动手

在开始做 2.1 和 2.2 之前，先把 extraction 这件事放慢一点：原来的想法是什么，今天真正推进了什么，以及接下来更稳的几条路。

今天本来准备继续往 wiki 的 extraction 细节里走，先做 2.1 和 2.2。

但在真正动手前，我们先把节奏放慢了一下。不是先写更多脚本，而是先去看：外面有没有已经很成熟的工具、框架、skill、plugin，能让这件事不要从零开始。

1. 最初的设想

最开始的想法其实很简单：

把历史材料慢慢抽成 wiki 能用的结构。

也就是从 HISTORY.md、以后更多 note / log / record 里，抽出：

  • 人
  • 项目
  • 工具
  • 地点
  • 事件
  • 关系
  • 证据

然后再把这些东西接进我们已经开始成形的 wiki kernel 里。

问题不在“能不能抽”，而在于：怎么抽得更稳，更可维护，也更适合以后变成一个长期系统。

2. 今天实际做了什么

今天没有直接继续写 extraction 主流程，而是先做了一轮外部调研和收敛。

重点看了几类东西：

  • 结构化抽取框架：Instructor、Outlines、BAML
  • 可优化的 LLM pipeline 框架：DSPy
  • 图谱 / graph extraction 方向：LlamaIndex Property Graph、GraphRAG
  • 有没有现成 plugin / skill 可以直接拿来做 personal wiki extraction

看完之后，结论反而更清楚了。

3. 今天得到的核心判断

现在最该找的，不是一个“万能插件”，而是一套分层方案。

因为我们的 extraction 问题，不是普通的表单抽取，也不是单次摘要。

它更像是：

  • 从私人历史材料里抽结构
  • 做规范化和去重
  • 保留 source / evidence
  • 最后进入本地 wiki kernel

这种事情太具体了，暂时没有看到一个现成 plugin 可以完整包办。

但好消息是，已经有足够成熟的积木可以借。

4. 目前最值得借的几块积木

  • Instructor：最适合做第一层结构化抽取，把 LLM 输出稳定成 schema。
  • LlamaIndex Property Graph 的思路：很适合指导 graph-shaped extraction 的设计。
  • BAML：如果后面 extraction pipeline 变复杂，适合做工程化编排。
  • DSPy：适合后期做 prompt / pipeline 优化，不是第一步。
  • GraphRAG：更像远期架构参考，提醒我们 extraction 不只是抽 entity，还要考虑 graph 和 summary。

5. 所以今天真正推进了什么

如果用最短的话总结，今天推进的不是“多写了一段 extraction 代码”，而是把方向收紧了：

  • 确认了 extraction 不应该只靠 ad hoc prompt
  • 确认了要用 schema-first 的方式思考
  • 确认了我们需要的是 kernel-oriented extraction，不是通用摘要
  • 确认了未来应该走 structured extraction + normalization + review + graph 这条路

这一步看起来慢，但其实是在避免后面走弯路。

6. 接下来可以怎么走

接下来有三条比较清楚的路。

选项 A：先做轻量 PoC

直接用 Instructor 接现在的 kernel schema，拿一小批 HISTORY.md 样本跑 extraction，先看结构和质量。

选项 B：先写 extraction design 文档

把候选工具、分层架构、数据模型、风险点、评测方式先写清，再决定 2.1 和 2.2 怎么落。

选项 C：先把 ontology / schema 再收紧一轮

先不碰模型，先把 nodes / relations / events / evidence / aliases 的边界定义得更稳定，再做抽取。

7. 我现在更倾向哪条路

如果只说今天结束时最稳的判断，我会倾向：

先 B，再 A，C 贯穿其中。

也就是：

  • 先把 extraction landscape 和设计写清
  • 再做一个很小的 PoC
  • 同时持续收紧 schema 和 ontology

这样 2.1 和 2.2 才不会只是“又多做了两步”，而是能接进一个以后真的能长起来的系统。

8. 先记到这里

今天更像一次停下来校方向，而不是猛推进。

但这种停一下，其实很值。

因为 extraction 一旦做成临时拼接，后面每加一个来源都会越来越乱。相反，如果一开始就按 schema、review、graph、local kernel 这几层来想，后面很多东西会自然一点。

先看清，再动手。

这就是今天最重要的进展。
